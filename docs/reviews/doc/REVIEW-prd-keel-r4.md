## MEDIUM 级问题

### [R-001] MEDIUM: F-012 AC-007「计划与额度」与定价延后决策存在歧义

- **category**: ambiguity
- **root_cause**: self-caused
- **描述**: F-012 AC-007 要求"显示当前消耗的计划与额度（透明度）"，但主卷 §4.1 明确"商业化定价方案不在本 PRD 范围（`ADR-0005`）"。「计划与额度」既可理解为 AI 执行引擎（Claude Code / Codex）的 API Token 消耗透明度，也可理解为 Keel 产品自身的付费计划用量，两种语义需要完全不同的实现（前者读取引擎侧 API，后者需要 Keel 自有计量系统）。当前措辞在定价延后的约束下含义不确定，架构师无法单一理解。
- **建议**: 在 F-012 AC-007 中明确「额度」的具体来源：若指执行引擎侧 API 使用量（如 Claude API Token 余额），则注明「显示执行引擎提供的 API 使用量信息」，并说明与定价决策无关；若超出 ADR-0005 延后范围，则暂时降为 [ASSUMPTION] 或移入后续 Sprint 范围。

### [R-002] MEDIUM: F-005 AC-004 自愈重试「上限」未定义

- **category**: ambiguity
- **root_cause**: self-caused
- **描述**: F-005 AC-004 写道"自愈重试达到上限仍失败时，升级为「需你决策」卡片"，但三卷任何位置均未给出具体的重试次数上限。「上限」是无可执行判据（oracle）的描述，测试者无法验证、架构师无法设计确定性的退出条件。F-003 AC-002 和 AC-004 同样提及「自动重试」但均未指定次数。
- **建议**: 在 F-005 AC-004 中给出具体数值（如"静默自愈最多重试 N 次（[ASSUMPTION] 默认 3 次，待架构阶段确认）"），或在 §4.3 关键假设中明确标注该数值为待定参数并提供候选范围。

### [R-003] MEDIUM: F-005 AC-002 技术/业务错误分类边界未可操作化

- **category**: ambiguity
- **root_cause**: self-caused
- **描述**: F-005 AC-002 将错误分为两类：「纯技术性错误（结构违规、类型错误、契约不符）」静默自愈，「业务决策类冲突（需要用户取舍的场景）」上浮为提示。但 AC 未给出分类判定规则，仅提供概念描述。对架构师而言，「需要用户取舍的场景」无法被确定性地检测，从而导致自愈层的实现逻辑无法被单测覆盖，无法判断分类器工作是否符合预期。
- **建议**: 在 AC-002 中补充分类准则，例如：「凡门禁错误具有唯一结构修复路径（如类型不符、接口签名不一致、依赖方向错误）视为纯技术性错误；凡涉及在两个合法蓝图选项间取舍（如两个功能模块边界冲突、同一能力多种实现方式）视为业务决策类冲突」，并可引用「[ASSUMPTION] 完整分类规则待架构阶段细化」。

### [R-004] MEDIUM: §3.1「主工作区初始加载≤2秒」无对应功能 AC

- **category**: completeness
- **root_cause**: self-caused
- **描述**: 主卷 §3.1 非功能需求性能表包含六条指标，其中五条均能在功能分卷中找到对应 AC（门禁 ≤5秒 → F-003 AC-005；能力地图渲染 ≤1秒 → F-006 AC-008；蓝图 diff ≤10秒 → F-009 AC-008；漂移检测 ≤3秒 → F-002 AC-005；时间线查询 ≤3秒 → F-013 AC-003）。但「主工作区初始加载（20节点地图）≤2秒」在三卷中没有对应功能 AC，无法被追溯和验收。
- **建议**: 在 F-006 或 F-008 中补充一条关于「主工作区初始加载」的 AC，明确加载范围（含能力地图 + 应用预览面 + 对话面的首次挂载时间），并引用 §3.1 的 2 秒目标值。

## LOW 级问题

### [R-005] LOW: F-010 AC-002 和 AC-004 是实现约束而非可观测 AC

- **category**: convention
- **root_cause**: self-caused
- **描述**: F-010 AC-002（"按依赖 DAG 拓扑序逐切片执行实现；每模块只注入自身契约 + 直接依赖签名的精准上下文"）和 AC-004（"实现在隔离 worktree 内进行"）描述的是内部架构约束，不是用户可观测的行为。用户无法感知执行顺序是否为拓扑序，也无法看到 worktree 是否隔离；这两条 AC 无法通过用户验收测试验证，只能通过代码或架构审查验证。这模糊了 PRD 与架构文档的边界。
- **建议**: 将 AC-002 和 AC-004 移入备注或重构为架构约束（如「[ASSUMPTION] 实现执行须以隔离单元操作，具体机制待架构阶段设计」），并在对应位置补充用户可感知的等效行为描述（如「单次能力实现失败不影响其他已完成切片」）。

### [R-006] LOW: F-011 缺少调和失败后的明确升级路径

- **category**: completeness
- **root_cause**: self-caused
- **描述**: F-011（调和引擎）的六条 AC 覆盖了正常调和流程，但未明确当调和本身反复失败时的升级路径。F-005 自愈机制和 F-010 AC-006「需你决策」卡片可通过推断覆盖此场景，但 F-011 本身无显式声明，导致实现者需要根据跨功能推断决定行为，增加理解歧义风险。
- **建议**: 在 F-011 备注中补充一句："调和执行期间同样适用 F-005 自动修复机制与 F-010 AC-006 升级路径；若调和反复失败无法自愈，同样升级为「需你决策」卡片。"

### [R-007] LOW: F-004 与 F-003 的用户升级路径不一致

- **category**: consistency
- **root_cause**: self-caused
- **描述**: F-003 门禁失败的处理链为：硬违规 → F-005 自动修复（静默自愈）→ 超限才升级「需你决策」。但 F-004 AC-002 写道：回归守护失败 → "硬阻断代码落地，并以通俗语言解释……「需要你决策」"，隐含直接升级用户而不经 F-005 自愈。这两条升级路径的差异未在文档中解释，架构师实现时需要自行判断回归失败是否经过自愈层。
- **建议**: 在 F-004 AC-002 备注或说明中明确：回归守护失败是否优先触发 F-005 自动修复，还是因为跨能力回归问题性质（涉及多切片冲突）而直接升级用户决策；若两者行为不同，须给出区分准则。

## 审查结论

**verdict**: approved_with_notes

全三卷无 CRITICAL 和 HIGH 问题。Layer 1 全部通过。壳级术语已清零，功能编号重排后交叉引用全部一致，P0 功能共 7/18（39%）且分级依据完整，主要流程文字描述与功能 AC 相互印证。

存在 4 条 MEDIUM 问题（R-001~R-004）和 3 条 LOW 问题（R-005~R-007），均为改善建议，不影响下游架构阶段启动。

其中 R-001（额度歧义）和 R-002（重试上限未定义）在架构阶段定义具体实现参数时必须解决；R-003（错误分类边界）和 R-004（NFR 孤立指标）建议在下一轮文档完善中处理；R-005~R-007 可在架构文档中承接处理。
