# Keel · 可用性测试方案 —— 验证 R1「可读」半边

> 本方案可直接拿去执行。被测对象是三张从 `blueprint.yaml` 渲染的原型:
> 原型 A 能力地图、原型 B 蓝图 diff 确认、原型 C 健康预警。
> 它验证的是 PRD 风险 **R1 的可读半边**(附录 B.11:「约束 ↔ 通俗表达」),
> 也是整个产品的 go / no-go 闸门。

---

## 0. 一句话目的

**让一群「重来者」型非程序员,在没人解释的情况下,看着这三张原型,用自己的话说清楚:
我的应用有哪些能力、它们怎么连;这次改动会改什么、影响谁;这条警告是什么问题、该怎么办。
能说清,R1 可读半边成立;说不清,地图/宪法的呈现方式必须重做。**

---

## 1. 目标与待验证假设

### 1.1 总目标
验证「声明式蓝图里的结构约束(依赖方向、契约、重复实现、影响范围、健康/漂移),
能否以能力地图 + 通俗解释的形式,被目标非程序员**无协助读懂**」。
强制半边(机器可检)已被认为基础扎实(B.11),本测试**只测可读半边**,不测好不好看、不测喜不喜欢。

### 1.2 精确到可读半边的三条子假设

| 编号 | 子假设(可读半边) | 对应原型 | 对应 PRD |
|---|---|---|---|
| H1 | 非程序员能从能力地图读懂「应用有哪些能力、它们怎么连(依赖关系)」 | 原型 A | FR-3 验收、§8、B.11 |
| H2 | 非程序员能从蓝图 diff 读懂「这次结构改动新增什么、影响哪些既有能力、不动什么」 | 原型 B | FR-4 验收「diff 解释不含代码术语,目标用户可读懂」、FR-8 |
| H3 | 非程序员能从健康预警读懂「这是什么结构问题、为什么要管、该怎么修」 | 原型 C | FR-6 验收「四类漂移以通俗语言展示」、流程 C |

> 三条子假设都成立 → R1 可读半边判定成立。任一条命中「重做线」(见 §6)→ 对应呈现需重做。

### 1.3 明确不在本测试范围
- 不测视觉审美偏好、不测「愿不愿意付费」、不测安装/登录/计费流程。
- 不测机器强制是否正确(那是另一条工程验证)。
- 不测真实代码生成质量(本测试不接引擎,原型是静态/半交互)。

## 2. 方法与形式

- **形式:远程主持式出声思考(moderated remote think-aloud)。** 参与者远程共享屏幕,主持人实时引导;经同意后录屏录音。
- **人数:5–8 名**,均须匹配 Persona A(见 §3)。可用性问题的边际发现率在 5 人后显著下降,5–8 人足以暴露这类「看不看得懂」的硬伤。
- **被试内设计、固定顺序 A → B → C。** 三张原型是一条连续叙事(先看应用 → 给它加「收藏」能力 → 几天后出健康预警),故**不打乱顺序、不做对照平衡**——这不是对照实验,而是三段递进的理解检验。在报告里注明顺序固定为有意设计。
- **单场约 45–60 分钟**:暖场与出声思考演示 → 任务①②③ → 收尾访谈。(此为单场协议时长,非工作量估算。)
- **每场一名主持 + 一名记录员**(记录员可由录屏回看替代,但实时记录更稳)。

### 2.1 测试材料
- 三个独立 HTML 文件:`prototype-A-capability-map.html`、`prototype-B-diff-review.html`、`prototype-C-health-warning.html`(本地起静态服务器或托管到内网/临时链接,参与者无需安装任何东西)。
- 唯一真相源 `blueprint.yaml`:仅供团队核对「界面内容 100% 来自蓝图、未从代码逆推」,**不给参与者看**。
- 录屏工具、同意书、记录表(§5)、计分卡(§4 各任务通过标准)。

### 2.2 关键守则:什么叫「无协助完成」
- 主持人可以澄清**任务要求**(「我想请你用自己的话说说这是什么意思」),
- 但**绝不解释界面内容、绝不说出答案或任何术语**。
- 参与者**自己**读到「通过」级答案 = 无协助完成(计为「通过」候选)。
- 经主持人中性提示(「你可以再看看别处」「有没有什么能点」)后才答对 = 「提示后完成」,**最高只能记「部分」**。
- 这条区分是整套判定的命门,务必在记录表逐题标注。

## 3. 参与者筛选(Persona A · 重来者)

### 3.1 纳入标准(全部满足)
1. **撞过墙:** 用过 Lovable / Bolt / v0 / Replit / Cursor 类「AI 帮我造应用」工具,且亲历过「项目变大后越改越乱 / 卡死 / 被迫重写」。
2. **有成长意图:** 当时想做的是「会持续加功能、有真实用户/数据」的东西,不是一次性玩具。
3. **半技术:** 能自己下载安装一个桌面应用并完成登录;但**不写代码、不会做架构设计**(关键词:「我能描述我要什么,但代码我看不懂」)。
4. **非开发者、非架构师**(排除职业程序员、技术负责人)。

### 3.2 排除标准
- 职业开发者 / 软件架构师 / 能独立读写 diff 与依赖图的人(他们「看得懂」不能证明非程序员看得懂)。
- 完全无法自行安装软件的纯小白(v1 不锁定该人群,见 PRD R4)。
- 与团队相熟、知道 Keel 设计意图的人(会污染「第一次看懂」)。

### 3.3 招募甄别问卷(5 题,口头或表单)
1. 你最近用过下面哪些工具做过东西?(多选:Lovable / Bolt / v0 / Replit / Cursor / 其它 / 都没用过 → 「都没用过」直接排除)
2. 你用它做的那个东西,后来怎么样了?(开放;听是否出现「越改越乱 / 卡住 / 重做」→ 命中纳入①)
3. 你本来是想做个长期能用、能不断加功能的东西,还是随手玩玩?(听「长期/加功能」→ 命中②)
4. 出问题的时候,你能自己看代码去改吗?(答「不能 / 看不懂代码」→ 命中半技术③;答「能,我会写」→ 排除)
5. 让你自己下载一个电脑软件、注册登录,你能搞定吗?(答「能」→ 命中半技术下限;答「不会装软件」→ 排除)

> 通过线:第 1 题非「都没用过」+ 第 2 题出现腐化经历 + 第 4 题「看不懂代码」+ 第 5 题「能装软件」。四条齐备方可纳入。

## 4. 三项核心任务

> 通用主持开场(每场一次,见 §4.0),随后按 ①→②→③ 进行。
> 每项给出:**场景搭建 / 提问话术 / 通过标准 / 失败信号 / 评分档**。

### 4.0 主持开场词(逐字念,降低紧张、建立出声思考)

> 「谢谢你来。今天大概 45 分钟到 1 小时。我想先说清楚:**今天测的是这个东西好不好懂,不是测你**——你看不懂,是它没做好,那正是我们想知道的,所以请放心说真话。
> 接下来我会给你看几张界面。每一张,我会请你**一边看一边把心里想的念出来**——你看到什么、觉得是什么意思、想点哪里、为什么。哪怕觉得'这不是废话吗'也请说出来。
> 我大多数时候不会回答你的问题,因为我想看的正是你**自己**会怎么理解它;你卡住了也没关系,卡住本身就是有用的信息。
> 这些界面是**早期草样**,有些地方点了可能没反应,那都正常,不影响今天。我们可以开始吗?」

随后做一次出声思考小演示(主持人对着任意网页演示念出自己的想法 20–30 秒),再进入任务①。

---

### 任务① —— 看懂「有哪些能力、它们怎么连」(原型 A · 对应 H1)

**场景搭建**
打开 `prototype-A-capability-map.html`。对参与者说:
> 「这是一个**已经做好的博客应用**,这张图是它现在长什么样的一张'地图'。先别急着点,你先随便看看。」

**提问话术(开放、不诱导,按顺序问)**
1. 「看完了用你自己的话告诉我:**这个应用现在能做哪些事?**」
2. 「图上这些方块之间有**线**连着。你觉得**这些线是什么意思?**」(切忌问「这是不是表示依赖?」)
3. 「随便挑一块点开看看,把里面写的念给我听。然后告诉我:**这一块跟别的有什么关系?**」
4. (进阶,可选)「假如'**登录**'这块以后出了毛病,你觉得**会不会连累到别的功能?为什么?**」

**通过标准(合格答案须包含,且为无协助)**
- 用通俗话点出 **≥3 个能力**(登录、文章列表、文章详情、评论中的三个);
- 把「线」解释成**两块能力有关联 / 一块用到(需要)另一块**(任意通俗说法均可,如「评论要先有登录」);
- 至少对**一块**能力,点开读出说明并**正确说出一条关系**(如「评论长在文章详情里」「评论得先登录」)。

**失败信号**
- 把方块当成「可以点进去使用的页面」,以为这就是在用真应用(把地图误当运行中的 App);
- 说不出线是什么意思,或解释成与依赖矛盾的东西(如「这是操作的先后步骤」并据此推错);
- 冒出「这是不是写代码的东西 / 我不懂编程看不了」之类被术语劝退的反应。

**评分档**
- **通过**:三条通过标准全部无协助达成。
- **部分**:能说出能力,但线的含义含糊/答错,或需提示才找到「点开看说明」。
- **失败**:说不清有哪些能力,或根本性误读地图(当成真 App / 认为线没意义)。

---

### 任务② —— 看懂「这次改动会改什么、影响谁」(原型 B · 对应 H2)

**场景搭建**
打开 `prototype-B-diff-review.html`。承接任务①的应用,对参与者说:
> 「还是刚才那个博客。你刚跟它说了一句话——'**我想让用户能收藏文章**'。它**还没动手**,先弹出这个给你看,让你确认。」

**提问话术**
1. 「用你自己的话说:它**打算给你的应用加什么?**」
2. 「这次会**碰到哪些已经有的东西?** 又有哪些它说**不会动?**」
3. 「图上那块**蓝色虚线的框**、还有那几条**蓝色虚线**,是什么意思?」
4. 「看完这些,你接下来会点'**确认**'、'**改一改**'还是'**放弃**'?**为什么?**」

**通过标准(无协助)**
- 说出这次是**新增一块「收藏 / 收藏文章」能力**(让人能收藏文章);
- 指出会被碰到的是**文章详情**和/或**登录**(至少 1 个正确),**且**说出**文章列表/评论不会动**(至少 1 个正确);
- 把蓝色虚线框/线理解为**「这次新加上去的东西」**;
- 能给出确认/改/放弃中的一个选择,**并说出与所读内容相符的理由**(不是随口选)。

**失败信号**
- 以为改动**已经做完了**(没意识到这是「动手前的确认」);
- 分不清**哪是新加的、哪是被影响的、哪是不动的**;
- 把「会碰到」理解成「会被弄坏 / 会被删掉」。

**评分档**
- **通过**:正确说出新增内容 + ≥1 个受影响 + ≥1 个不动 + 看懂虚线=新增,且能做出有依据的决定,全程无协助。
- **部分**:看懂「加了收藏」,但受影响/不动的部分含糊,或需提示。
- **失败**:误以为已完成,或无法区分 新增/受影响/不动。

---

### 任务③ —— 看懂「这条健康警告是什么问题、该怎么办」(原型 C · 对应 H3)

**场景搭建**
打开 `prototype-C-health-warning.html`。对参与者说:
> 「'收藏'功能加好之后,过了几天,你打开应用,发现**顶上的灯变红了**,出现了这个画面。」

**提问话术**
1. 「用你自己的话说:**现在出了什么问题?**」
2. 「图上**哪一块**出问题了?你**怎么看出来的?**」
3. 「这个问题**为什么需要管它?不管会怎样?**」
4. 「如果你想**解决**它,你会**怎么做?**」
5. (找到「一键合并」后)「点了'一键合并'之后,你**预期会发生什么?**」

**通过标准(无协助)**
- 用通俗话说出问题本质:**「收藏」被做了两遍 / 有两套在打架**(重复实现);
- 指出出问题的是**收藏文章**这一块,并说明依据(红点/红框/读到的文字);(能顺带分辨出「评论」那条是不着急的小毛病 = 加分,非必需);
- 说出一条**合理的「为什么要管」**(会越改越乱 / 以后会出毛病,哪怕表述粗糙);
- **找到并理解修复入口「一键合并」**,知道这是用来解决它的(且预期是系统去合并、不用自己写代码)。

**失败信号**
- 说不出问题是什么,或读成「收藏功能坏了 / 用不了了」(问题性质错);
- 把红色的那块和「问题」对不上,找不到是哪儿;
- 找不到该怎么办,或以为得自己去改代码 / 去别处弄。

**评分档**
- **通过**:说出「做了两遍/重复」+ 定位到收藏那块 + 给出合理的「为什么要管」+ 找到并理解一键合并,全程无协助。
- **部分**:看懂「收藏这块出了点事 + 找到了修复按钮」,但说不清是「做了两遍」,或需提示。
- **失败**:误读问题性质,或定位不到、不知道该怎么办。

## 6. 判定闸门(go / no-go 阈值)

> 记 N = 实际样本数(5–8)。**只有「无协助达到通过标准」才计入「通过」。**

### 6.1 单任务三条线

| 线 | 判据 | 含义 |
|---|---|---|
| **通过线** | 「通过」人数 ≥ ⌈70% × N⌉ **且** 「失败」人数 ≤ 1 | 该呈现方式**验证通过**,可继续投入。 |
| **重做线(硬触发)** | 「通过」人数 < 50% × N **或** 「失败」人数 ≥ ⌈1/3 × N⌉ | 该呈现方式**必须重做**后再测。 |
| **中间区** | 介于两线之间 | **有条件通过**:针对失败信号做**定向修订**(不必推倒),小步迭代后补测。 |

**N=6 的具体数字(便于现场判定):**
- 通过线:**≥5 人通过 且 ≤1 人失败**。
- 重做线:**≤2 人通过 或 ≥2 人失败**。
- 其余(如 3–4 人通过、0–1 人失败)= 中间区,定向修订。

### 6.2 产品级 R1 可读半边判定

- **R1 可读半边成立(go):** 任务①②③ **全部** 达到各自「通过线」。→ 地图/diff/健康的呈现方向被验证,继续投入。
- **R1 可读半边不成立(no-go,需重做呈现):** **任一**任务命中「重做线」。
  - 若命中的是**任务①(能力地图本身)**:这是最严重情形——**能力地图与宪法的呈现方式需整体重新设计**,因为②③都建立在「能读懂地图」之上。
  - 若命中的是②或③:对应的 diff 呈现 / 健康呈现需重做,地图主呈现可保留。
- **有条件 go:** 三项均未命中重做线,但存在中间区任务。→ 按 §7 输出定向修订项,迭代后对该项补测,达到通过线即转 go。

> 这条闸门即 PRD 所说「若目标用户读不懂,地图与宪法的呈现方式必须重做」的可执行化。

## 7. 主持注意事项(防止把测试做废)

1. **测「看懂」,不测「喜欢」。** 不要问「你喜欢吗 / 好不好看 / 顺不顺手」;只问「这是什么意思 / 你会怎么做 / 为什么」。审美和偏好评价记到备注,**不计分**。
2. **绝不诱导。** 不要把答案塞进问题里。
   - ✗「这条线是不是表示依赖关系?」 ✓「这条线你觉得是什么意思?」
   - ✗「红色是不是说明有问题?」 ✓「你看到这个画面,第一反应是什么?」
   - 不用点头、皱眉、「嗯对」「不是哦」等暗示对错的反应;用中性的「嗯,继续说」。
3. **不替参与者解释。** 参与者卡住时:先**沉默等待**(心里数 5–10 秒,沉默是允许的);仍卡再给**中性提示**(「你可以再看看别的地方」「这上面有没有什么是能点的」);**永远不说出答案、不说任何术语**。每次提示都原样记到记录表——给过提示的题最高只能记「部分」。
4. **保持出声思考。** 参与者沉默时轻推:「你现在在想什么?」「你为什么想点这儿?」
5. **任务之间不「教」。** 做完①不要纠正对错,否则会污染②③的「第一次看懂」。
6. **固定顺序、一次一人。** 不打乱 ①→②→③;不群测。
7. **允许放弃单题。** 参与者明确放弃 = 该题记「失败」,不强求。
8. **主持人自己别下场答题。** 你越想帮忙,数据越假。你的沉默就是最好的工具。
