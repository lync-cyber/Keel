Keel 蓝图与工作机制

一份给团队对齐用的图文说明：把 PRD 里「蓝图」「编译」「门禁」这些概念，从抽象原则落成可看懂的机制图。作为后续迭代设计的初始基线。

基于 demo/blueprint.yaml（示例应用「我的博客」v7） · 2026-06-09
一份蓝图，编译出两个产物：一半给机器强制结构，一半给人读懂结构——这是 Keel 全部机制的主轴。

1蓝图是什么

蓝图（Blueprint）是整个应用的唯一真相源。它声明式地描述「应用应该长什么样」——有哪些能力、怎么分层、谁依赖谁、数据是什么、对外契约如何——而代码必须永远服从蓝图。

关键反转：传统做法是「代码是真相，文档去追代码」；Keel 反过来——蓝图是真相，代码去追蓝图。任何结构上的偏离（破边界、重复实现、孤立代码、契约不符）都叫漂移，会被门禁挡住或被健康预警标红。这正是「不腐化」的来源。

声明式：描述期望态，不写步骤 版本化：每次变更可追溯（本示例 v7） 机器可检：每条约束都能被检查器执行 通俗可读：非程序员能从地图读懂
R1 — 产品成立的核心假设：蓝图能否同时做到「机器可检」与「非程序员可读」。机器可检这半边基础扎实；可读半边尚未用真实用户验证，是当前最大风险，也是 go / no-go 闸门。下面第 2 节的「双表达」就是为这半边而生。

2双表达结构：一份约束，两种说法

蓝图里每个构造都同时带技术字段（给机器/检查器）和 plain_* 字段（给非程序员）。两者源于同一处，永不分叉——这是 R1 可读半边的设计落点。原型界面只渲染 plain_*，技术字段一律不上界面。

例 A · 一条依赖规则

技术字段（机器可检）

id: no-cross-feature-import
formal: feature 之间不得直接 import；
跨能力交互只能经 core 或对方 contract.ts
severity: hard

plain 字段（非程序员可读）

每个功能各管各的，不许偷偷伸手进别的功能内部拿东西；要打交道得走正门（对方挂出来的对外接口）。

例 B · 一个能力

技术字段（机器可检）

id: favorite
module: src/features/favorite
contract: .../contract.ts
entities: [Favorite]
depends_on: [article-detail, auth]

plain 字段（非程序员可读）

收藏文章 — 让登录的人把喜欢的文章收起来。收藏按钮长在文章详情页上，要先登录才能用。
依赖原因也用通俗话写：「收藏按钮长在文章详情页上，没有文章详情就没处放收藏按钮。」

3蓝图的构造（用「我的博客」看）

一份蓝图由这几类构造组成。每个 capability（能力） 同时对应：代码里一个切片 features/<能力>/、地图上一个节点——三者一一对应（PRD 附录 B.1）。

meta 应用名 / 原型 / 版本 / 技术栈 layers ui · logic · data core 登录管家 · 数据管家 dependency_rules 4 条结构纪律 entities 用户 · 文章 · 评论 · 收藏 capabilities 5 个能力
实线 = 已有依赖（A 依赖 B：A 离不开 B）；蓝色虚线与虚线框 = 本次 diff 新增的「收藏文章」及其依赖。颜色 = 健康态（绿=健康）。
线代表依赖，不是操作步骤：箭头从「用到别人的」指向「被用到的」。例如「评论 → 文章详情 / 登录」表示评论既长在文章详情里、又要先登录才能发。

4依赖方向规则（结构纪律）

蓝图里有 4 条「违反即硬阻断」的依赖规则（PRD 附录 B.3）。它们是让结构不漂移的骨架——新能力只能在这套规则内生长。

no-cross-feature-import 功能间不直接 import，走正门 layer-direction ui→logic→data 单向 data-via-core 数据访问必经 core one-canonical-impl 一个职责只能一处实现

5工作机制主轴：编译

编译器是把「声明式蓝图」变成「机器可执行门禁 + 人类可读地图」的确定性过程。这是承重主轴——后面的门禁、漂移、调和全都消费它的产物。

产物①是标准、厂商中立的工具配置，会提交进用户仓库。这意味着即使脱离 Keel，导出的项目仍能自我强制结构——这就是「可迁出、不锁定」的技术落点（FR-11）。意见被物化成标准配置，而不是专有运行时。

7三层组合与合并

「有效蓝图」不是一份手写文件，而是三层确定性合并出来的（PRD 附录 B.10）。优先级显式；存在一组密封不变量，覆盖层碰它即编译期拒绝。

这套分层让 Keel 升级原型默认时不冲掉用户的覆盖（类似 eslint/tsconfig 的 extends 继承），同时保证「不可谈判的结构」始终被锁住。D3 已拍板：v1 只上 1 个内置原型「特性切片 / 模块化单体」。

8门禁运行模型

编译出的门禁配置在每次代码改动处运行（经 ACP 权限中介或引擎 hooks）。硬违规阻断、软问题放行，确保 LLM 既不漂移、也不被逼到空转。

🔒 隔离：每次能力改动在独立 worktree「实现→门禁→校验→合并」 🔁 自动修正：硬违规先自愈，超预算才升级 🛟 需你决策：最后安全阀，不静默卡死

9漂移检测与调和

漂移检测复用同一套编译产物，只是从「编辑时拦截」改成「对全仓持续 / 增量扫描」，把偏离归成四类，汇成健康红绿灯 + 通俗问题清单。

破边界 · dependency-cruiser 重复实现 · ts-morph 孤立未接线 · 可达性分析 契约不符 · tsc / schema

示例：本应用当前健康报告（红）

灯问题通俗说明 → 怎么修
红「收藏」被做了两遍两处代码都在管收藏、在打架（违反 one-canonical-impl，硬）→ 一键合并，系统自动收拢成一处。
黄「评论」有个名字没按规矩起不影响使用，只是命名不一致（软）→ 一键规范命名。小乱子积少成多才会让结构看不清。

调和 = 检测到漂移或用户改了蓝图时，驱动 LLM 定点修正使代码重新服从蓝图；过程同样受门禁约束、有界、不触发整体重写。

10端到端叙事：给博客加「收藏」

把上面所有机制串成一条用户真会走的路。三张原型（A 地图 / B diff / C 健康）正是这条叙事的三个切面。

  1. 说一句话：用户在对话面板说「我想让用户能收藏文章」。（状态：CAPTURING）
  2. 翻成蓝图 diff：系统产出 diff——新增能力 favorite + 实体 Favorite，标注受影响（文章详情、登录）与不动（文章列表、评论）。地图上以蓝色虚线呈现。→ 原型 B（状态：DIFF_REVIEW）
  3. 确认后受门禁实现：在 worktree 内驱动引擎写代码，每次编辑过门禁。能力节点从「规划中→实现中→已就绪」。（状态：IMPLEMENTING）
  4. 几天后出漂移：代码里冒出两处都在做收藏 → 漂移检测判 one-canonical-impl 重复实现，健康灯转红。→ 原型 C
  5. 一键调和：用户点「一键合并」，受门禁定点修正，把两处收拢成一处；回归校验通过，健康灯恢复绿，记入意图日志。
全程用户没看一行代码、没碰终端。他看到的只是：地图、通俗 diff、健康灯、一键按钮——这正是 R1 要验证的「可读半边」能否独立成立。

11名词速查

名词一句话
蓝图 Blueprint唯一真相源，声明式描述应用应有的样子；代码必须服从它。
能力地图蓝图的人类可读视图，非程序员的主界面，永远从蓝图渲染、不从代码逆推。
双表达每个构造同时有技术字段（机器）和 plain_*（人）；界面只渲染后者。
编译把蓝图确定性地变成「门禁配置 + 地图视图」两个产物。
门禁 Gate每次改动处运行检查器、违规即拦截的确定性机制。
漂移 Drift代码对蓝图的偏离：破边界 / 重复 / 孤立 / 契约不符。
调和 Reconcile把漂移定点收敛回期望态，有界、不整体重写。
架构原型 Archetype「默认结构+宪法+门禁规则集+检查器配置」的捆绑，由意图自动选中。
