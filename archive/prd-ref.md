## 产品功能定义与需求总结

### 1. 产品定位

**一句话**：把软件架构从"会过期的文档"升级为"可执行、可验证、可视化的图中间表示（IR）"，让 LLM 在图约束下生成代码，让人在图层面（而非代码层面）理解、审查和掌控软件。

**解决的问题**：LLM 编程的结构性幻觉（虚构 API、违反依赖、重复造轮子）、长期架构腐化与漂移、人类审查海量生成代码的心智负担。

**能力边界（明确声明）**：机械化拦截结构性错误（接口、依赖、分层、契约）；逻辑正确性由图驱动生成的测试兜底，不做过度承诺。

### 2. 总体架构：一核两壳

一个 headless 内核引擎 + 两个面向不同市场的产品壳，GTM 单线主攻、先后推出，可双品牌。

### 3. 内核需求（共享，约 60–70% 工程量）

- **双层图 IR**（day-one 不可逆决策）：业务层（领域概念）+ 实现层（软件结构），层间映射为一等公民；文本/JSON 双形态，进 Git 可 diff；开放格式，目标成为事实标准，agent 无关
- **七个原语**：Module、Interface（带类型契约）、Entity/DataModel（含所有权）、Dependency（含允许/禁止规则）、Workflow/Sequence、StateMachine、Constraint（不变量、NFR 预算、安全规则）
- **双向同步引擎**（最大工程纵深与护城河）：代码→图逆向提取（AST 级静态分析）、设计图与实际图的持续 drift 检测、偏差双向裁决
- **验证门**：编译 + 契约测试 + 架构 lint 三道门；失败输出结构化错误供自动重试
- **多视图投影**：同一 IR 投影为 C4 分层视图、序列图、状态机图、数据流图——给 PM 看 Context 层、给开发看 Component 层
- **LLM 编排**：按依赖 DAG 拓扑序调度，每模块只注入自身契约 + 直接依赖签名的精准上下文
- **表达力陷阱与逃生舱**。 图原语永远无法表达全部需求（性能调优、复杂算法、UI 细节）。需要设计 escape hatch：标注为"自由实现区"的模块、附加自然语言意图说明的节点。
- **基础设施**：安全扫描、版本与一键回滚、escape hatch（标注"自由实现区"的模块）

### 4. 壳 A：开发者版（嫁接 Claude Code，MCP/插件/CLI 形态）

**三档渐进式介入**（每档独立成立）：
1. **只读档**（获客钩子）：`arch map` 一条命令 30 秒生成存量仓库的可交互活架构图 + 漂移/腐化体检报告；可分享链接、README 徽章
2. **守护档**：CI 架构检查、PR 图级 diff、影响半径（blast radius）高亮、图级 approve
3. **生成档**：需求对话→图草稿→人审图确认→骨架生成→约束下拓扑序填充

**关键交互**：活图（点节点跳代码、保存秒级更新、节点叠加健康信号）；LLM 工作时的"空中管制塔"视图（节点实时点亮/变红）；越界裁决卡（违反什么/影响多大/接受偏差更新图 或 拒绝重做，留痕）；验证失败对人隐形、对 LLM 透明自动重试；大图层级聚合 + 任务焦点子图；零配置 recommended 约束集，自定义为高级功能。

**MVP 范围**：TypeScript 单语言、模块化单仓、四原语（Module/Interface/Dependency/Constraint）、brownfield 逆向优先。

### 5. 壳 B：小白版（Lovable 模式，Web 产品）

**定位**：不做第 N 个"想法到应用"，做"应用到资产"——Day-30 复杂度墙之后的市场：看得懂、改不坏、敢上线。

**原语全量翻译为业务语言**：功能区块 / 应用记住了什么 / 当…就… / 订单的一生 / 安全规则；零软件术语。

**核心功能**：
- 三栏界面：聊天 + 应用地图 + 预览；地图默认极简、按需展开、可完全只用聊天
- 指图说话（point-and-prompt）：选中节点再提需求
- 改动前影响预告："会碰这 2 个区块，不会碰支付和客户数据"
- 人话变更说明书：图 diff + 自然语言解释取代代码 diff
- 上线门禁：认证/数据权限/密钥的红绿灯安全体检与安全分
- 时光机：一键回滚到任意地图状态
- **自动修复代理层**（小白版独有厚层）：消费内核结构化错误静默自愈，仅业务决策才用人话上浮
- 锁定技术栈（固定托管/数据库/认证），收窄解空间保证质量
