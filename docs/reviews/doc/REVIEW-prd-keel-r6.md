## Layer 2 — 三处变更语义审查

### 变更 1：F-018 AC-006 新增（prd-keel-f010-f018.md）

**变更内容**：在 F-018「执行引擎抽象层」新增 AC-006，全文为：「Keel 核心能力（门禁检查、蓝图读取/校验、能力地图状态、漂移报告）以执行引擎可直接消费的**工具面**暴露：候选形态含 MCP Server、引擎 hooks、CLI 命令，与 ACP 权限中介并列互补；具体形态组合待架构阶段选型（[ASSUMPTION]）。无论选型结果如何，执行引擎在实现/调和过程中必须能以结构化方式调用门禁并读取 JSON 结果（衔接 F-003 AC-006）。」

**备注内容**：「AC-006 的工具面是**产品内部接缝**（面向执行引擎/LLM 消费），不构成面向开发者用户的独立工具产品，与 `ADR-0011` 的范围边界不冲突。」

**审查维度**:

**完整性 (completeness)**：
- AC-006 覆盖了工具面的四类核心能力（门禁检查、蓝图读取/校验、地图状态、漂移报告），清单是否完整？—— 与 F-018 其他 AC 的能力范围（ACP 接缝、SDK 回退）对比，工具面四项对应执行引擎在实现/调和过程中的实际消费需求，合理且有据可查。
- 最小必然要求「以结构化方式调用门禁并读取 JSON 结果」有明确表述，保留了架构选型空间同时锁定了底线行为。

**衔接引用一致性 (consistency)**：
- AC-006 末尾写「衔接 F-003 AC-006」——核查 F-003 AC-006（见分卷 1）是否确实描述了门禁结构化输出。分卷 1 的 F-003 AC-006 是「需你决策」的人工决策路径，**并非**门禁结构化 JSON 输出的定义处。

  这里存在引用目标与描述意图的偏差：「衔接 F-003 AC-006」的措辞暗示 F-003 AC-006 定义了「门禁 JSON 结果格式」，但 F-003 AC-006 的实际内容是人工升级路径（「门禁反复硬阻断且自动修复无法推进时，以通俗语言说明冲突点…用户决策留痕」）。门禁结构化输出（JSON 结果）的定义并不在 F-003 AC-006，而更可能是 F-003 的其他 AC 或架构阶段设计。

  此引用会让架构师产生歧义：F-003 AC-006 是人机界面定义，不是机器间接口定义。

- F-018 AC-001~AC-005 与新 AC-006 的关系：AC-001~AC-004 描述了 ACP 接缝与解耦；AC-006 以「与 ACP 权限中介并列互补」表述，逻辑定位清晰，无内部矛盾。

**歧义性 (ambiguity)**：
- 「工具面」一词在 F-018 AC-006 和 §3.3 均出现，但 PRD 术语表（§6）无对应定义。对于后续架构阶段，「工具面」是否等价于「MCP Tool 接口」、「CLI subcommand」、「hooks 事件接口」存在多义。由于 AC-006 本身列了候选形态并标注 [ASSUMPTION]，歧义程度可接受，但补充术语表定义可降低理解成本。
- 「能力地图状态」是否等同于 F-006 定义的 Capability Map 渲染数据，还是仅指节点状态（`planning/implementing/ready`）？措辞略宽泛，但标注了 [ASSUMPTION] 架构选型，架构阶段可确认，不构成阻塞性问题。

**范围合规性（ADR-0011 一致性）**：
- 备注明确「产品内部接缝（面向执行引擎/LLM 消费），不构成面向开发者用户的独立工具产品，与 ADR-0011 不冲突」——措辞与 ADR-0011 后果段的「边界澄清」高度互补，自洽。
- 「MCP Server」在 AC-006 中作为候选形态之一，结合 ADR-0011 原文「开发者工具产品线（MCP/CLI/IDE 插件接入形态）移出本 PRD 范围」，有表面张力。但 ADR-0011 移出的是「面向开发者的 MCP 接入形态（开发者作为最终用户）」，而 AC-006 中的 MCP Server 是 Keel 内部暴露给执行引擎的机器消费接缝。两者确实不同：一个是产品形态，一个是内部技术机制。ADR-0011 后果段新增的「边界澄清」条目进一步确认了这一区分，三处变更联合形成自洽体系。
- 不存在变相恢复开发者产品线的风险，因为 AC-006 无任何面向人类开发者的交互面或文档化 API 义务。

### 变更 3：ADR-0011 后果段「边界澄清」（docs/decisions/0011-focus-non-programmer-product.md）

**变更内容**：在「后果」段最后追加「边界澄清：移出的是「人类开发者作为最终用户」的产品形态；执行引擎（LLM）消费 Keel 门禁/蓝图/地图能力的**内部工具接缝**（ACP / MCP Server / hooks / CLI）属产品内部机制，保留在范围内（见 PRD F-018 AC-006）。」

**审查维度**:

**一致性 (consistency)**：
- 「边界澄清」与 F-018 AC-006 备注互为镜像：F-018 AC-006 的备注声明「不与 ADR-0011 不冲突」，ADR-0011 新增条目声明「内部工具接缝保留在范围内（见 F-018 AC-006）」。双向引用构成完整的可追溯链，一致性高。
- 「移出的是「人类开发者作为最终用户」的产品形态」是对 ADR-0011 正文「面向开发者的工具产品线」的精确语义窄化，未改变原决策语义，只是澄清边界，符合 amendment 的 clarification 性质。

**歧义性 (ambiguity)**：
- ADR-0011「后果」段的结构从「变更列表」变成「变更列表 + 边界澄清」，逻辑上「边界澄清」是对「决定」段文字的解释性附加，不是一条后果条目。当前放置位置合理，不引入结构歧义。

**范围合规性**：
- 「ACP / MCP Server / hooks / CLI」与原 ADR-0011 正文中「MCP/CLI/IDE 插件接入形态」的列表存在交叉（MCP 和 CLI 同时出现在「移出」和「保留」的表述中），但语义通过「人类开发者作为最终用户 vs. 执行引擎消费」做了精确切割。此切割逻辑清晰，不构成自相矛盾，但确实需要读者仔细分辨两处 MCP/CLI 指的不同层次。

## MEDIUM 级问题

### [R-001] MEDIUM: F-018 AC-006 引用 F-003 AC-006 目标不准确

- **category**: consistency
- **root_cause**: self-caused
- **描述**: AC-006 末尾写「衔接 F-003 AC-006」，但 F-003 AC-006（分卷 1）的实际内容是「需你决策」卡片的人工决策升级路径，并非「门禁结构化 JSON 输出」的定义处。架构师读到此引用时，会前往 F-003 AC-006 寻找「机器可读 JSON 结果」的定义，却找到的是人机界面升级路径，产生语义错配。
- **建议**: 将引用改为指向 F-003 中实际定义门禁结构化输出的 AC（若存在），或改为更宽泛的「衔接 F-003 门禁检查器」并在 [ASSUMPTION] 中注明「JSON 结果格式规范待架构阶段定义」。如果 F-003 暂无对应 AC，可删除此引用，改为「执行引擎在实现/调和过程中必须能以结构化方式调用门禁并读取 JSON 结果（[ASSUMPTION] 具体 JSON schema 待架构阶段定义）」，避免空悬引用。

## LOW 级问题

### [R-002] LOW: §3.3 与 F-018 AC-006 措辞轻微不统一

- **category**: convention
- **root_cause**: self-caused
- **描述**: 主卷 §3.3 用「LLM 可消费的工具面」，分卷 F-018 AC-006 用「执行引擎可直接消费的工具面」，备注用「执行引擎/LLM 消费」。三处措辞指向同一语义，但用词不完全一致，轻微增加跨章节对齐的认知负担。
- **建议**: 可在术语表（§6）新增「工具面（Tool Surface）」条目，统一表述为「面向执行引擎（LLM）的机器消费接缝」，并将三处行文措辞统一引用此定义；或在 §6 已有「执行引擎」词条旁附注工具面关系。非阻塞性，架构师可在架构文档中统一。

### [R-003] LOW: ADR-0011 MCP/CLI 在「移出」和「保留」中双重出现，增加读者辨别成本

- **category**: ambiguity
- **root_cause**: self-caused
- **描述**: ADR-0011 正文「决定」段列有「MCP/CLI/IDE 插件接入形态」为移出范围，新增的「边界澄清」条目又将「MCP Server / hooks / CLI」列为保留在范围内。两处使用相同技术词汇（MCP、CLI）但指不同层次（产品形态 vs. 内部机制），读者需要仔细阅读上下文才能辨别，存在初次阅读时产生「决策矛盾」的感知风险。
- **建议**: 在「边界澄清」条目中，将「ACP / MCP Server / hooks / CLI」改为「ACP / MCP Server（作为内部 tool 服务） / engine hooks / CLI 脚本」等更具描述性的表述，与正文「MCP/CLI/IDE 插件接入形态（面向开发者的产品入口）」在措辞上形成可见区分，降低混淆概率。非阻塞性修改。
