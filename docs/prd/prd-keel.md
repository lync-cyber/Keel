## 1. 概述

### 1.1 背景与动机

非程序员的 AI 编程已规模化，但存在共性困境：用户能快速起步，项目一旦变大就开始腐化——AI 越改越乱、用户无法读懂代码加以补救，最终卡死或被迫重写。开发者侧已有成熟的防腐实践（spec-driven development、结构门禁），但都面向开发者，非程序员无法直接使用。

Keel（龙骨）是一个面向非程序员的单一产品：让不会写代码的人零代码构建「可持续成长且不腐化」的 Web 应用。它把软件架构从「会过期的文档」升级为「可执行、可验证、可视化的图中间表示（IR）」，让 LLM 在图约束下生成代码，让人在图层面（而非代码层面）理解、审查和掌控软件，并保证代码全程归用户所有、可随时导出迁走。

Keel 的核心主张是**应用蓝图作唯一真相源 + 确定性门禁防结构漂移**。蓝图是声明式、版本化、机器可检的期望态，门禁是不可绕过的确定性检查器，两者共同保证「结构不腐化、契约一致」。

**能力边界（明确声明）**：机械化拦截结构性错误（接口、依赖、分层、契约）；逻辑正确性由图驱动生成的测试兜底，不做过度承诺。无可执行判据（oracle）的约束（产品意图、交互体验等）回退人工批准锚定。对可写出可执行判据（oracle: test/property）的约束，远期纳入行为门（契约派生单测/属性测试）；当前阶段只做字段预留（`ADR-0009`）。

**竞争定位**：

- 对 Lovable / Bolt / v0 / Replit：它们擅长起步、却在成长期腐化；Keel 主打持续迭代不腐化。
- 对 Floot / Mocha：它们以闭环锁定用户；Keel 开放代码、不锁定、支持迁出。
- 对 Spec Kit / Kiro / CC-SDD：它们要求会写 spec、读 diff；Keel 把同等结构纪律做成非程序员可用。

### 1.2 目标用户

Keel 是面向非程序员的单一产品，功能清单不按用户群或产品形态分版（`ADR-0011`）。

**Persona A · 重来者（非程序员，主用户）**

- 用过 Lovable / Bolt 等 AI 编程工具，亲历过越改越崩，成长意图已被验证。
- 半技术：能安装桌面应用并完成登录，能读懂能力地图与业务语言的蓝图视图，但不会做架构。
- 核心诉求：这次不要再崩，而且代码归我。

**Persona B · 接手的开发者（间接角色）**

- 不直接使用 Keel；日后通过导出物与交接包承接 Persona A 用 Keel 建造的项目。
- 其上手体验是「可迁出」目标的验收对象（F-015 / F-016）。

### 1.3 成功指标

| 指标 | 目标值 | 衡量方式 |
|------|--------|----------|
| 核心假设验证（R1）：非程序员能读懂蓝图与能力地图 | 测试参与者中 ≥70% 能据地图回答「我的应用有哪些能力、它们如何连接」 | Web 原型可用性测试（`ADR-0002`）|
| 需求与验收准则可读性（R1）| 测试参与者中 ≥70% 能读懂蓝图中的 plain 字段与 acceptance 条目 | 同上 |
| 门禁有效性 | 任何违反蓝图边界的改动均被拦截，零漏报 | 结构性违规集成测试 |
| 代码可迁出性 | 导出物在无 Keel 环境下可独立运行，结构约束由标准工具持续强制 | 导出后独立运行验证 |

## 3. 非功能需求

### 3.1 性能

| 场景 | 指标 | 目标值 |
|------|------|--------|
| 门禁检查单次执行 | 单次检查器运行时间 | ≤ 5 秒 |
| 能力地图首次渲染（20 个节点以内） | 渲染时间 | ≤ 1 秒 |
| 主工作区初始加载（20 节点地图） | 加载时间 | ≤ 2 秒 |
| 蓝图 diff 生成（意图翻译后） | 响应时间 | ≤ 10 秒 |
| 漂移检测（增量，单次改动后） | 检测时间 | ≤ 3 秒 |
| 时间线历史查询 | 响应时间 | ≤ 3 秒 |

### 3.2 安全

- 鉴权凭据（API Key / OAuth Token）本地存储，最小权限授权，不上传至 Keel 服务端。
- 蓝图文件含结构约束，不含业务数据或用户个人信息。
- 上线门禁须覆盖认证配置、数据权限、密钥泄露三类安全项，输出红绿灯安全分（F-014）。
- 开放代码导出物不得包含凭据或内部服务地址硬编码。

### 3.3 兼容性

- 目标产物栈：Web 全栈 JavaScript/TypeScript（前端 React/Next.js 类，后端 Node.js/Serverless 类，关系型数据库）。
- [ASSUMPTION] 代码生成目标在 v1 仅支持 TypeScript 单语言、模块化单仓；多语言支持为后续项。
- 执行引擎接入：优先 ACP 中立协议（Claude Code / Codex 经 adapter 接入），回退 Agent SDK 或无头 CLI；Keel 核心能力另以执行引擎可直接消费的工具面暴露（MCP Server / hooks / CLI 候选，F-018 AC-006）。
- Web 原型预览：现代主流浏览器（Chrome/Firefox/Edge 最新稳定版）；移动端浏览器不在 MVP 范围。

## 4. 约束与假设

### 4.1 产品约束

- **范围约束**：单人 · 单项目 · Web 全栈 JS/TS · 结构不腐化 · 可导出迁走。多人协作、跨项目看板、可复用资产市场、自建部署基础设施、移动端及非 Web 应用类型均不在范围内。
- **单一产品边界**：面向开发者的工具产品线（存量仓库 arch map、CI/PR 架构检查、面向开发者用户的 MCP/CLI/IDE 插件产品入口）不在本 PRD 范围（`ADR-0011`）；执行引擎消费 Keel 能力的内部工具接缝不受此限（F-018 AC-006）。如未来启动开发者产品线须另立独立 PRD，不与本产品功能清单混编。
- **锁定技术栈**：固定托管提供商 / 数据库 / 认证方案，收窄解空间，保证代码生成质量可控；不支持用户自选技术栈（[ASSUMPTION] 具体锁定选项待架构阶段确认）。
- **非目标**：撞墙解套（已腐化存量项目的修复）、跨工作流（写作、运营等非应用构建场景）。
- **定价延后**：商业化定价方案不在本 PRD 范围，留待 R1 假设验证后决策（`ADR-0005`）。

### 4.2 技术约束

- **唯一真相源**：`docs/blueprint.example.yaml` 是规范蓝图实例（符合 `docs/blueprint.schema.json`）；`demo/blueprint.yaml` 已弃用（`ADR-0007`）。
- **界面内容来源**：界面只渲染 `plain_*` 字段，技术字段一律不上界面；界面内容改动须核对 100% 来自蓝图、未从代码逆推。
- **v1 单一架构原型**：v1 只上 1 个默认原型 `modular-monolith`（特性切片/模块化单体），由意图自动选中（`ADR-0003`）。
- **行为门远期纳入**：当前门禁只保证结构不腐化、契约一致；行为门（契约派生单测/属性测试）为 R1 之后的远期项，现在只做 `oracle` 字段占位预留（`ADR-0009`）。
- **门禁严重度分层**：结构性违规（破依赖边界、重复实现、契约不符）为硬阻断；风格类问题（命名、目录、轻微冗余）为软告警（`ADR-0004`）。
- **蓝图文件格式**：探索阶段使用 YAML + JSON Schema；不上 CUE/Pkl，等到编译强制时再评估（`ADR-0010`）。
- **需求层（R-IR）**：蓝图含 `requirements[]` 原语，带 `id / plain / acceptance[]`（含 oracle 标注）；`Unit` 与 `Constraint` 通过 `satisfies` 回链实现需求覆盖率校验（`ADR-0008`）。

### 4.3 关键假设

- **核心假设 R1**（须优先安排最小验证）：蓝图能否同时做到机器可检与非程序员可读。这是产品成立的前提。
- [ASSUMPTION] 自动修复代理层（消费门禁结构化错误静默自愈）的自愈策略与上浮判定逻辑待架构阶段设计，本 PRD 只声明功能意图（F-005）。
- [ASSUMPTION] LLM 编排的按依赖 DAG 拓扑序调度，精准上下文注入，具体调度协议待架构阶段确定。

### 4.4 调研记录

- 产品统一决策：`ADR-0011`（聚焦非程序员单一产品，开发者工具产品线移出范围）
- Web 原型先行决策：`ADR-0002`（先用现有三张 Web 原型测试核心假设，嵌入式为后续项）
- 参考技术报告：「形式化输入 → 确定性变换规则 → 可验证输出」三元式（见 `archive/meta-model.md` §1）

## 5. 关键流程

### 流程 A · 主循环：加一个能力

1. 用户在对话面板用通俗语言描述需求；也可先选中地图节点再描述（指图说话，F-009）。
2. 系统产出蓝图 diff，在地图上以高亮与差异标注呈现，并给通俗解释与受影响能力清单（「会碰这 N 个区块，不会碰…」）。
3. 用户确认、追加修改（回到第 2 步）、或拒绝。
4. 确认后先冻结本次涉及的接口契约与数据 schema，再驱动引擎针对冻结契约逐切片实现；每次编辑经过门禁，硬违规即拦截并自动重试，软问题记录健康。
5. 进度以能力地图节点状态变化展示（规划中 → 实现中 → 已就绪），不向用户暴露代码或终端输出。
6. 实现完成后运行回归守护与漂移检测，健康灯更新，记入意图日志。

**异常分支**：门禁反复硬阻断且无法自动推进时，升级为「需你决策」卡片，用通俗语言说明冲突点并给出选项（F-010）。

### 流程 B · 新项目上手

1. 新建项目，检测执行引擎；缺失则图形化安装引导，经应用登录完成鉴权，全程无终端命令。
2. 描述应用意图或选起步模板，自动选定 `modular-monolith` 架构原型。
3. 生成初始蓝图与可运行骨架，能力地图首次渲染。
4. 引导用户加第一个能力，进入流程 A。

### 流程 C · 健康预警与调和

1. 健康灯转黄或红，问题清单出现通俗描述的具体项（如「收藏功能被实现了两遍，建议合并」）。
2. 用户点选某项，显示通俗说明与修复入口。
3. 触发调和：以受影响切片的冻结契约为锚，驱动引擎整片重生成该切片（非就地修补），受门禁约束。
4. 修复后回归校验通过，健康灯恢复，记入意图日志。

### 流程 D · 一键上线

1. 用户点击上线，系统运行安全体检：认证配置 / 数据权限 / 密钥泄露三类红绿灯检查，输出安全分。
2. 存在红灯时阻断上线，逐项给出通俗说明与修复入口（可触发流程 C）。
3. 全绿后部署到锁定技术栈的托管环境，向用户返回可访问地址。
4. 上线动作记入意图日志，时间线可回滚。

### 流程 E · 导出与交接

1. 用户在项目设置选择导出代码或生成交接包。
2. 导出标准 JS/TS 仓库（含编译落地的标准工具配置 dependency-cruiser / eslint / tsconfig / schema），脱离 Keel 仍能自我强制结构。
3. 生成开发者交接包：蓝图转架构文档、意图日志转决策史、边界规则转上手指南。

## 6. 术语表

| 术语 | 定义 |
|------|------|
| 应用蓝图（Blueprint） | 唯一真相源。声明式、版本化、机器可检的期望态，描述应用应有的样子；代码必须永远服从蓝图。 |
| 能力地图（Capability Map） | 蓝图的人类可读视图，是用户的主界面，永远从蓝图渲染，不从代码逆推。 |
| 漂移（Drift） | 代码与蓝图的偏离，包括破边界、重复实现、孤立未接线代码、契约不符。 |
| 调和（Reconcile） | 以受影响切片的契约为锚、整片重生成使其收敛回期望态的有界过程；不触发整体重写，也不在既有代码上就地修补。 |
| 门禁（Gate） | 在每次代码改动处运行检查器、违规即拦截的确定性机制。分硬阻断（结构性违规）和软告警（风格类）。 |
| 执行引擎（Engine） | 实际驱动代码改动的底层 agent（Claude Code / Codex），由 Keel 无头驱动。 |
| 架构原型（Archetype） | 一套捆绑的「默认结构 + 宪法 + 门禁规则集 + 检查器配置」，v1 默认原型为 `modular-monolith`。 |
| 图 IR（Graph IR） | 应用结构的图中间表示（双层：业务层 + 实现层），是蓝图的内部表达形式，文本/JSON 双形态，进 Git 可 diff。 |
| R-IR（需求 IR） | 蓝图中的需求原语层（`requirements[]`），带 EARS 句式、可测验收准则与 oracle 标注。 |
| 契约冻结 | 意图确认后、实现开始前，将本次涉及的接口契约与数据 schema 锁定为只读锚点。 |
| 切片（Slice） | 按能力边界划分的代码单元（`features/<capability>/`），调和与实现均以切片为边界操作。 |
| 指图说话（point-and-prompt） | 选中地图节点后再输入需求，系统将意图绑定到该节点，降低歧义。 |
| 时光机 | 可回滚时间线的产品化名称：以自然语言描述历史时间点，一键回滚到任意地图状态，不暴露 git 概念。 |
| 安全体检 | 上线门禁的产品化名称：认证配置 / 数据权限 / 密钥泄露三类红绿灯检查，红灯阻断上线。 |
| plain_* 字段 | 蓝图中供界面渲染的通俗字段（`plain`、`plain_name` 等），技术字段不上界面。 |
| oracle | 验收准则的判定方式：`test / property / type / schema / human`；`human` 表示无可执行 oracle，回退人工批准。 |
| blast radius | 某次变更会影响到的能力与模块集合；在地图上高亮展示。 |
| escape hatch | 标注为「自由实现区」的模块，允许附加自然语言意图说明，绕过严格图原语表达。 |
