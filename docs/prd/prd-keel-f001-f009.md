## 2. 功能需求

### F-001: 蓝图核心原语与元模型

- **用户故事**: 作为用户，我希望系统用一套统一的结构原语来描述我的应用，以便蓝图能同时被机器执行检查、又能被我（非程序员）读懂。
- **验收标准**:
  - [ ] AC-001: 蓝图支持七个原语：Module/Unit（有边界的构件）、Interface/Port（对外接口）、Entity/DataModel（持久化数据）、Dependency/Relation（含允许/禁止规则）、Workflow/Sequence（交互流程）、StateMachine（状态机）、Constraint（不变量/NFR/安全规则）。
  - [ ] AC-002: 每个原语必须携带 `plain` 字段（通俗表达），技术字段不上界面；`plain` 字段空缺时，系统拒绝渲染该原语到能力地图。
  - [ ] AC-003: 蓝图支持 `requirements[]` 原语（R-IR）：每条需求含 `id`、`plain`、`acceptance[]`（可测验收准则，含 `oracle` 标注）；`Unit` 与 `Constraint` 可通过 `satisfies` 字段回链需求 id。
  - [ ] AC-004: 编译期校验：每条 `requirements[]` 条目至少被一个 `Unit` 或 `Constraint` 的 `satisfies` 覆盖（需求覆盖率为 100% 时校验通过）。
  - [ ] AC-005: `Constraint.severity` 枚举为 `hard / soft`：`hard` 触发硬阻断，`soft` 流过软告警。
  - [ ] AC-006: `Port` 支持可选 `contract`（`pre / post / invariant`，子句 `plain` 必填、`formal` 可选），为远期行为门预留占位；当前只校验格式合法性，不执行行为测试。
  - [ ] AC-007: 规范实例 `docs/blueprint.example.yaml` 为唯一权威蓝图实例，符合 `docs/blueprint.schema.json`；`demo/blueprint.yaml` 已弃用，顶部含弃用说明。
- **优先级**: P0（分级依据：七原语是整个产品的基础数据结构，无此则蓝图无法定义，下游所有功能无从运行；P0 的「没有则产品不可用」条件完全成立）
- **备注**: 七原语中 Workflow/Sequence 和 StateMachine 为 v1 最小化支持，完整语义待架构阶段设计（[ASSUMPTION]）。行为门（`oracle: test/property` 实际执行）为远期项（`ADR-0009`），v1 只做字段预留。

### F-002: 蓝图双向同步与漂移检测

- **用户故事**: 作为用户，我希望系统能自动检测代码与蓝图之间的偏离，以便我无需手动对比代码就能知道应用是否还在「正轨」上。
- **验收标准**:
  - [ ] AC-001: 代码变更后，系统能通过 AST 级静态分析从代码逆向提取当前结构，并与设计蓝图持续对比，检测出漂移。
  - [ ] AC-002: 漂移检测覆盖四类：破依赖边界、重复实现同一功能、孤立未接线代码、接口契约不符。
  - [ ] AC-003: 漂移以通俗语言展示（如「收藏功能被实现了两遍，建议合并」），不向用户暴露代码术语。
  - [ ] AC-004: 双向裁决：漂移既可以「回收代码，服从蓝图」（触发调和 F-011），也可以「更新蓝图，承认实际」（触发蓝图 diff 确认流程 F-009），两种路径均支持。
  - [ ] AC-005: 增量检测：单次改动的漂移检测自启动至完成 ≤ 3 秒（本指标仅约束检测执行时长；通道外改动从发生到被系统感知的发现延迟不计入，由 AC-006 的发现时机选型决定）。
  - [ ] AC-006: 漂移检测**与变更来源无关**：无论代码变更经由 Keel 驱动的执行引擎产生，还是在 Keel 之外被直接修改（用户或第三方工具直接编辑文件），均被同等检测；不存在因「改动未经 Keel 通道」而漏检的路径。验收方式：在 Keel 之外直接修改文件分别制造 AC-002 四类漂移，漂移检测均须报告。（[ASSUMPTION] 外部改动的发现时机——文件系统监听实时触发，或项目打开/下次操作时扫描——待架构阶段设计；无论选型为何，不得削弱本条的来源无关语义。）
- **优先级**: P0（分级依据：双向同步是「不腐化」核心价值主张的感知层，无此则漂移无法被检测，产品无法提供「知道应用是否偏轨」的核心承诺）
- **备注**: AST 级静态分析实现策略（工具链、语言支持范围）待架构阶段选型（[ASSUMPTION]）。v1 只支持 TypeScript（`ADR-0003` 延伸）。本功能与验证门构成双层防御：F-003 是阻断性门禁（通道内落地前事前拦截，不合规代码不落地），F-002 是检测性机制（对全量实际态持续对账，reconcile loop 模式，发现已存在的偏离但本身不阻断），承担通道外改动的兜底检测，检出漂移后经 F-011 调和收敛。

### F-003: 验证门（Gate）

- **用户故事**: 作为用户，我希望每次代码生成后系统自动校验结构合规性，以便不合规的代码永远无法落地。
- **验收标准**:
  - [ ] AC-001: 验证门包含三道：编译门（TS 类型检查）、契约测试门（Port 接口一致性校验）、架构 lint 门（依赖方向、分层、重复实现规则）。
  - [ ] AC-002: 硬违规（`Constraint.severity: hard`）时门禁阻断代码落地，并输出结构化错误消息供 LLM 自动重试。
  - [ ] AC-003: 软告警（`Constraint.severity: soft`）时门禁记录健康问题，代码照常落地，问题在能力地图健康面板可见。
  - [ ] AC-004: 验证失败消息对用户隐形、对 LLM 透明自动重试；用户只看到「正在修复…」状态，不暴露错误日志或代码。
  - [ ] AC-005: 单次门禁检查执行时间 ≤ 5 秒。
  - [ ] AC-006: 门禁通过统一的结构化接口触发，检查结果为机器可解析的 JSON 格式，支持 LLM 读取结构化错误后自动重试；具体接口协议待架构阶段确定（[ASSUMPTION]）。
- **优先级**: P0（分级依据：验证门是「确定性门禁防结构漂移」这一核心卖点的执行机制，无此则门禁为空壳，「结构不腐化」的承诺无法兑现）
- **备注**: 行为门（契约派生单测 / 属性测试）为远期项（`ADR-0009`），当前门禁只保证结构不腐化、契约一致。AC-002 / AC-004 所述「自动重试」的次数上限统一由自动修复层（F-005 AC-004）管理，本功能不单独定义重试参数。

### F-004: 回归守护

- **用户故事**: 作为用户，我希望每次加新功能后，系统自动验证之前的功能没被破坏，以便我不会在不知情的情况下引入回归。
- **验收标准**:
  - [ ] AC-001: 随能力增加累积行为检查，每次改动后运行（与验证门 F-003 结合执行）。执行模型分层：F-003 负责本次改动的结构合规门（单点检查）；F-004 在 F-003 通过后追加跨能力回归校验（累积检查），两者串行执行，结果统一上报到能力地图健康状态。
  - [ ] AC-002: 新改动导致既有能力检查失败时，硬阻断代码落地，并优先交由自动修复层（F-005）尝试自愈（修复新改动使其不再破坏既有能力，与 F-003 硬违规的升级路径一致）；自愈无法收敛、或修复方案涉及业务取舍（如新旧两个能力的需求本质冲突）时，以通俗语言上浮「这次改动破坏了 [能力名]，需要你决策」。
  - [ ] AC-003: 回归守护结果在能力地图上可见：已通过的能力节点保持绿色，被破坏的节点变红并带说明。
  - [ ] AC-004: 会破坏既有能力的改动会被拦截并展示说明，而非静默落地。
- **优先级**: P1
- **备注**: 回归守护的「行为检查」在 v1 阶段基于契约测试与结构检查，不包含端到端行为测试（行为门为远期项，`ADR-0009`）。

### F-005: 自动修复（静默自愈）

- **用户故事**: 作为非程序员，我希望系统遇到技术性错误时自动修好它，只有需要我做业务决策时才来问我，以便我永远不用面对看不懂的技术错误消息。
- **验收标准**:
  - [ ] AC-001: 自动修复层消费门禁（F-003 / F-004）输出的结构化错误，驱动执行引擎静默自愈；自愈期间用户只看到「正在修复…」状态。
  - [ ] AC-002: 错误分类上浮判定准则：凡门禁错误存在唯一结构性修复路径（类型不符、接口签名不一致、依赖方向错误、契约不符）视为纯技术性错误，静默自愈，且修复不得改变已确认蓝图 diff 的语义；凡修复需在多个合法蓝图方案间取舍、或会改变已确认蓝图语义（增删改用户已确认的能力、数据或边界）视为业务决策类冲突，上浮为通俗提示（[ASSUMPTION] 完整分类规则待架构阶段细化，但「是否改变已确认蓝图语义」为不变判据）。
  - [ ] AC-003: 用户在任何情况下都不会看到原始技术错误消息（堆栈、编译输出、lint 日志）。
  - [ ] AC-004: 静默自愈对同一错误最多重试 3 次（[ASSUMPTION] 默认值，待架构阶段确认调参）；达到上限仍失败时，升级为「需你决策」卡片（F-010 AC-006），不无限重试、不静默放弃。
- **优先级**: P1
- **备注**: 自愈策略与上浮判定逻辑待架构阶段设计（[ASSUMPTION]，见主卷 §4.3）。本功能是产品体验护城河：缺失时门禁错误将以原始形式频繁上浮，非程序员体验显著下降，但核心构建流程仍可用，故 P1 而非 P0。

### F-006: 能力地图（Capability Map）

- **用户故事**: 作为非程序员，我希望有一张活的应用地图，以便我能随时看懂我的应用有哪些能力、它们如何连接，而无需读代码。
- **验收标准**:
  - [ ] AC-001: 地图内容 100% 来自蓝图 `plain_*` 字段渲染，不从代码逆推；任何「界面内容」改动须在变更记录中核对来源为蓝图。
  - [ ] AC-002: 地图以「能力—模块—数据—连接」呈现应用形状，全程通俗表达，不暴露代码术语（类名、文件路径等）。
  - [ ] AC-003: 七原语全量翻译为业务语言呈现：功能区块（Module）/ 应用记住了什么（Entity）/ 当…就…（Workflow）/ 订单的一生（StateMachine）/ 安全规则（Constraint）；界面零软件术语。
  - [ ] AC-004: 地图叠加健康状态（绿/黄/红）；红色节点可点击进入问题详情与修复入口。
  - [ ] AC-005: 点击节点可查看该能力的说明、所属模块、依赖关系，全程通俗语言。
  - [ ] AC-006: 地图只读；结构变更一律走意图捕获流程（F-009），地图上无直接编辑入口。
  - [ ] AC-007: 大图支持层级聚合（折叠子图）与任务焦点子图（只展示当前变更相关的节点）；地图默认极简、按需展开。
  - [ ] AC-008: 能力地图渲染时间（20 节点以内）≤ 1 秒。
- **优先级**: P0（分级依据：能力地图是非程序员理解和掌控应用的唯一界面，无此则用户无法在图层面理解、审查自己的应用，产品核心主张「人在图层面掌控软件」无法成立）
- **备注**: 多视图投影（同一 IR 投影为序列图、状态机图、数据流图等技术视图）面向接手开发者与交接场景，v1 仅在交接包（F-016）中以文档形式体现，不进用户主界面（[ASSUMPTION]）。

### F-007: 改动影响预览（Blast Radius）

- **用户故事**: 作为用户，在我提出需求之前，我希望看到这次改动会影响哪些功能区块，以便我有把握做决定，不会误伤其他部分。
- **验收标准**:
  - [ ] AC-001: 基于蓝图依赖图，在地图上高亮本次变更将触及的能力节点与模块边界（blast radius）。
  - [ ] AC-002: 影响预告以通俗语言表达：「会碰这 N 个区块：[区块名列表]，不会碰 [安全区块名列表]」。
  - [ ] AC-003: 预览所示影响范围与实际改动范围一致，无明显漏标（漏标率为 0）。
  - [ ] AC-004: 影响预告在蓝图 diff 确认界面中同步展示，不需要额外操作。
- **优先级**: P1
- **备注**: blast radius 计算依赖蓝图依赖图的完整性；依赖图有环或信息缺失时降级展示（[ASSUMPTION]）。

### F-008: 应用实时预览

- **用户故事**: 作为非程序员，我希望在构建过程中随时看到我的应用真实长什么样、能不能用，以便我用「亲眼所见」而非技术指标来判断进度和效果。
- **验收标准**:
  - [ ] AC-001: 主工作区提供应用实时预览面：展示当前应用的真实运行界面，可直接交互操作。
  - [ ] AC-002: 实现完成（门禁与回归守护通过）后预览自动刷新到最新状态；刷新无需用户手动操作。
  - [ ] AC-003: 预览不可用时（应用骨架未生成、构建中）展示通俗状态说明，不出现技术错误页。
  - [ ] AC-004: 主工作区同屏提供对话、能力地图、应用预览三个交互面；用户可完全只用对话面完成构建（地图与预览为增强理解，不是操作前提）。
  - [ ] AC-005: 主工作区初始加载（含对话面、20 节点能力地图、应用预览面的首次挂载）≤ 2 秒（对应主卷 §3.1）。
- **优先级**: P1
- **备注**: 三个交互面的具体布局与响应式行为属 UI 设计范畴，由 ui-spec 定义；本 PRD 只约定「同屏可用」与各面职责。

### F-009: 意图捕获与蓝图 diff

- **用户故事**: 作为用户，当我说「加一个用户能收藏文章的功能」，我想先看到它会怎样改动应用结构，确认无误再让它动手。
- **验收标准**:
  - [ ] AC-001: 自然语言输入产出蓝图 diff：新增或修改哪些能力、触及哪些边界、定义什么契约。
  - [ ] AC-002: 以「地图 diff + 通俗解释」呈现（人话变更说明书）：用户看到的是「添加了收藏功能，修改了用户信息区块」类自然语言描述，而非代码 diff；解释不含代码术语。
  - [ ] AC-003: 指图说话（point-and-prompt）：用户可先选中地图节点再输入需求，系统将意图绑定到该节点理解（「针对这个区块的意图」），降低歧义。
  - [ ] AC-004: 改动前影响预告展示 blast radius（与 F-007 复用核心逻辑）：「会碰这 N 个区块，不会碰支付和客户数据」类通俗表述。
  - [ ] AC-005: 用户可确认、追加自然语言修改（回到 diff 生成）、或拒绝；三条路径均支持。
  - [ ] AC-006: 确认后先冻结本次涉及的接口契约（`Port` 签名）与数据 `Entity` schema，作为实现阶段的只读锚点；冻结本身是一道独立门，契约不完整或不一致则退回澄清。
  - [ ] AC-007: 任何改变结构的请求都必须先产出蓝图 diff 并获确认，不存在绕过蓝图直接写代码的路径。
  - [ ] AC-008: 蓝图 diff 生成响应时间 ≤ 10 秒。
  - [ ] AC-009: 每次变更绑定意图日志条目（含原始意图文本与对应蓝图 diff），支持后续追溯（F-013）。
- **优先级**: P0（分级依据：意图捕获是用户与系统交互的核心入口，无此则用户无法发起任何变更；「先产出蓝图 diff 再确认」是防止直接写代码绕过约束的唯一屏障）
- **备注**: 无。
