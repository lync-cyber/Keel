## 增量修复核验

### R-001 修复核验：F-012 AC-007 额度歧义

<!-- 原问题：「计划与额度」既可理解为 AI 执行引擎 API Token 消耗透明度，也可理解为 Keel 自身付费计划用量，两种语义完全不同 -->

**修复位置**: 分卷 2，F-012 AC-007

**修复前（r4）**: 「显示当前消耗的计划与额度（透明度）」（模糊）

**修复后（r5）**: 「显示执行引擎侧的订阅计划与用量信息（如 Claude Code / Codex 订阅或 API Key 的额度与消耗，数据由引擎侧接口提供）；该展示仅为执行引擎消耗透明度，与 Keel 自身商业化定价无关（定价延后，`ADR-0005`）。」

**核验结论**: 修复有效。新措辞明确了三个关键点：(1) 信息来源为执行引擎侧接口；(2) 范围为引擎订阅/API Key 消耗；(3) 与 Keel 自身定价无关并引用 ADR-0005。架构师可据此单一理解，不存在歧义。

**结果**: RESOLVED

### R-002 修复核验：F-005 AC-004 重试上限未定义

<!-- 原问题：AC-004 写「达到上限仍失败」但未给出具体次数 -->

**修复位置**: 分卷 1，F-005 AC-004；F-003 备注

**修复后 AC-004**: 「静默自愈对同一错误最多重试 3 次（[ASSUMPTION] 默认值，待架构阶段确认调参）；达到上限仍失败时，升级为「需你决策」卡片（F-010 AC-006），不无限重试、不静默放弃。」

**修复后 F-003 备注**: 「AC-002 / AC-004 所述「自动重试」的次数上限统一由自动修复层（F-005 AC-004）管理，本功能不单独定义重试参数。」

**核验结论**: 修复有效。F-005 AC-004 给出了可操作数值「3 次」，并正确标注 [ASSUMPTION] + 架构待确认。F-003 备注的补充消除了重试次数在两处各自定义的歧义风险，「统一由 F-005 管理」声明清晰。

**结果**: RESOLVED

### R-003 修复核验：F-005 AC-002 错误分类边界未可操作化

<!-- 原问题：「纯技术性错误」与「业务决策类冲突」只有概念描述，无判定规则 -->

**修复位置**: 分卷 1，F-005 AC-002

**修复后**: 「错误分类上浮判定准则：凡门禁错误存在唯一结构性修复路径（类型不符、接口签名不一致、依赖方向错误、契约不符）视为纯技术性错误，静默自愈，且修复不得改变已确认蓝图 diff 的语义；凡修复需在多个合法蓝图方案间取舍、或会改变已确认蓝图语义（增删改用户已确认的能力、数据或边界）视为业务决策类冲突，上浮为通俗提示（[ASSUMPTION] 完整分类规则待架构阶段细化，但「是否改变已确认蓝图语义」为不变判据）。」

**核验结论**: 修复有效。新 AC-002 做到了三点：(1) 给出了具体的结构性修复路径举例（类型不符/接口签名不一致/依赖方向错误/契约不符）；(2) 立「是否改变已确认蓝图语义」为不变判据，提供了可被单测覆盖的二元决策标准；(3) [ASSUMPTION] 标注承认细化待定，边界明确。与 r4 建议高度一致。

**结果**: RESOLVED

### R-004 修复核验：§3.1「主工作区初始加载≤2秒」无对应功能 AC

<!-- 原问题：六条性能指标中「主工作区初始加载 ≤2秒」无功能 AC 与之对应 -->

**修复位置**: 分卷 1，F-008 新增 AC-005

**修复后 AC-005**: 「主工作区初始加载（含对话面、20 节点能力地图、应用预览面的首次挂载）≤ 2 秒（对应主卷 §3.1）。」

**核验结论**: 修复有效。新 AC-005 明确了加载范围（含三个面的首次挂载），给出了 2 秒目标值，并通过「对应主卷 §3.1」建立了双向可追溯链接。与 r4 建议完全对齐。

**结果**: RESOLVED

### R-005 修复核验：F-010 AC-002/AC-004 是实现约束而非可观测 AC

<!-- 原问题：AC-002（DAG 拓扑序执行）和 AC-004（worktree 隔离）是内部架构约束，用户无法感知和验收 -->

**修复位置**: 分卷 2，F-010 AC-002、AC-004、备注

**修复后 AC-002**: 「以已冻结契约为锚，按蓝图依赖关系排序逐切片实现：被依赖的切片先就绪，用户在地图上可观察到节点按依赖顺序进入「已就绪」。」

**修复后 AC-004**: 「实现过程隔离：单次能力实现失败或中止时，应用保持本次改动前的完整可用状态，已就绪的其他切片不受影响、不留半成品。」

**修复后备注**: 「执行引擎集成协议见 F-018（[ASSUMPTION] 具体协议待架构阶段设计）。实现机制约束（架构承接，非用户可观测验收项）：按依赖 DAG 拓扑序调度、每模块仅注入自身契约与直接依赖签名的精准上下文、隔离 worktree 内「实现—校验—合并」（[ASSUMPTION] 具体机制待架构阶段设计，AC-002 / AC-004 验收其用户可观测效果）。」

**核验结论**: 修复有效。AC-002 改写为「用户在地图上可观察到节点按依赖顺序进入『已就绪』」，AC-004 改写为「单次失败不影响其他已就绪切片、不留半成品」，均为用户可验收的外部行为。原内部实现约束（DAG 拓扑序、精准上下文注入、worktree 隔离）正确移入备注，并标注为「架构承接，非用户可观测验收项」。PRD 与架构文档的边界得到清晰划定。

**结果**: RESOLVED

### R-006 修复核验：F-011 缺少调和失败后的明确升级路径

<!-- 原问题：F-011 无显式声明调和反复失败的处理路径 -->

**修复位置**: 分卷 2，F-011 备注

**修复后备注**: 「调和与重写的边界判定（切片边界如何划定）依赖架构阶段对 `Unit` 粒度的设计（[ASSUMPTION]）。调和执行期间同样适用 F-005 自动修复机制（含其重试上限）与 F-010 AC-006 升级路径：调和反复失败无法自愈时，升级为「需你决策」卡片，不静默放弃。」

**核验结论**: 修复有效。备注明确了两条升级链：(1) 适用 F-005 自动修复机制（含重试上限）；(2) 升级 F-010 AC-006「需你决策」卡片；(3) 兜底条款「不静默放弃」。覆盖了 r4 建议的完整升级路径。

**结果**: RESOLVED

### R-007 修复核验：F-004 与 F-003 的用户升级路径不一致

<!-- 原问题：F-004 AC-002 隐含直接升级用户，而 F-003 先经 F-005 自愈再升级，两条路径差异未解释 -->

**修复位置**: 分卷 1，F-004 AC-002

**修复后 AC-002**: 「新改动导致既有能力检查失败时，硬阻断代码落地，并优先交由自动修复层（F-005）尝试自愈（修复新改动使其不再破坏既有能力，与 F-003 硬违规的升级路径一致）；自愈无法收敛、或修复方案涉及业务取舍（如新旧两个能力的需求本质冲突）时，以通俗语言上浮「这次改动破坏了 [能力名]，需要你决策」。」

**核验结论**: 修复有效。新 AC-002 明确了路径：硬阻断 → F-005 自愈 → 自愈不收敛或涉及业务取舍 → 上浮用户。括注「与 F-003 硬违规的升级路径一致」进一步消除了跨功能理解分歧。给出了「自愈无法收敛」和「涉及业务取舍」两个清晰的上浮触发条件。

**结果**: RESOLVED
