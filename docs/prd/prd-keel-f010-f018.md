## 2. 功能需求

### F-010: 受门禁约束的实现执行

- **用户故事**: 作为用户，在我确认了变更计划后，我希望系统自动完成代码实现，全程无需我接触代码或终端，且每步都有确定性检查防止跑偏。
- **验收标准**:
  - [ ] AC-001: 仅在契约冻结通过后（F-009 AC-006）进入实现阶段；实现期间 `Port` 签名、目录结构与 `Entity` schema 为只读，不可被 LLM 修改。
  - [ ] AC-002: 以已冻结契约为锚，按蓝图依赖关系排序逐切片实现：被依赖的切片先就绪，用户在地图上可观察到节点按依赖顺序进入「已就绪」。
  - [ ] AC-003: 每次文件改动经验证门（F-003）校验；硬违规即拦截，经自动修复层（F-005）静默自愈；软问题记录健康。
  - [ ] AC-004: 实现过程隔离：单次能力实现失败或中止时，应用保持本次改动前的完整可用状态，已就绪的其他切片不受影响、不留半成品。
  - [ ] AC-005: 进度以能力地图节点状态变化呈现（规划中 → 实现中 → 已就绪），用户不看到代码或终端输出。
  - [ ] AC-006: 「需你决策」卡片：门禁反复硬阻断且自动修复无法推进时，以通俗语言说明冲突点（违反什么 / 影响多大）并给出选项（接受偏差更新蓝图 / 拒绝并重做）；用户决策留痕到意图日志（F-013）。
  - [ ] AC-007: 一次正常的加能力实现中，用户无需看到任何代码或终端命令。
- **优先级**: P0（分级依据：实现执行是从意图到代码落地的唯一路径，无此则意图捕获（F-009）只能产出 diff 但无法落地，产品无法完成任何功能构建）
- **备注**: 执行引擎集成协议见 F-018（[ASSUMPTION] 具体协议待架构阶段设计）。实现机制约束（架构承接，非用户可观测验收项）：按依赖 DAG 拓扑序调度、每模块仅注入自身契约与直接依赖签名的精准上下文、隔离 worktree 内「实现—校验—合并」（[ASSUMPTION] 具体机制待架构阶段设计，AC-002 / AC-004 验收其用户可观测效果）。

### F-011: 调和引擎（Reconcile）

- **用户故事**: 作为用户，当系统检测到代码腐化（漂移），我希望能一键修复，而不需要手动改代码或理解技术细节。
- **验收标准**:
  - [ ] AC-001: 调和以受影响切片为单位，以该切片的冻结契约为锚，驱动引擎整片重生成 `features/<capability>/` 切片，不在既有代码上就地修补。
  - [ ] AC-002: 重生成以切片为边界，无状态进行：每次仅依据该切片的契约与本地依赖，不积累跨切片漂移。
  - [ ] AC-003: 调和过程同样受验证门（F-003）约束；重生成的切片必须通过门禁才能落地。
  - [ ] AC-004: 调和后该项漂移消失（漂移检测不再报告该项），且不引入新的回归（回归守护 F-004 通过）。
  - [ ] AC-005: 调和为有界操作，以切片为边界，不触发整体重写。
  - [ ] AC-006: 调和入口：用户在健康问题清单点选某项后，系统展示通俗说明并提供一键修复按钮。
  - [ ] AC-007: 整片重生成前的内容保护：当待调和切片含 `escape_hatch: true` 模块（F-017），或含未被冻结契约/图原语完整表达的行为（如仅 `plain` 占位、无 `formal` 契约的逻辑），调和**不得静默整片重写覆盖**——须先以通俗语言提示"这块有你的自由实现区/未入图的内容，重做可能丢失"，并要求用户确认后再执行；escape hatch 区可选择"仅提示不重生成"。
  - [ ] AC-008: 重生成行为等价校验：调和重生成切片后，须经回归守护（F-004）验证既有能力不回归；若重生成导致既有验收行为失败，按 F-010 AC-006 升级为「需你决策」卡片，不静默落地行为退化的切片。
- **优先级**: P1（分级依据：调和是修复已检测到腐化的执行手段，但产品在「能检测漂移、展示健康问题、阻断新硬违规」的状态下仍具核心价值；调和缺失时用户须手动修复，体验下降但产品不失基本可用性。相比之下 F-001/F-002/F-003 等任一缺失均导致产品核心流程断裂，调和不满足 P0 标准）
- **备注**: 调和与重写的边界判定（切片边界如何划定）依赖架构阶段对 `Unit` 粒度的设计（[ASSUMPTION]）。调和执行期间同样适用 F-005 自动修复机制（含其重试上限）与 F-010 AC-006 升级路径：调和反复失败无法自愈时，升级为「需你决策」卡片，不静默放弃。

### F-012: 项目初始化与上手

- **用户故事**: 作为重来者（非程序员），我希望下载安装一个应用、点一下登录，就能开始描述我的应用，全程不碰终端、不手抄密钥。
- **验收标准**:
  - [ ] AC-001: 检测本地执行引擎（Claude Code / Codex 桌面应用、CLI 或 ACP adapter）是否就绪；未安装时给出图形安装引导，不出现终端命令。
  - [ ] AC-002: 鉴权优先触发应用自带的订阅登录（OAuth）；仅持有 API Key 的用户亦兼容，本地存储、最小权限授权。
  - [ ] AC-003: 经 ACP 中立协议或 Agent SDK / 无头 CLI 在后台连接执行引擎，并跑冒烟任务校验连通；桌面图形界面仅用于安装与登录，不用于驱动。
  - [ ] AC-004: 用户描述应用意图（自由输入或选起步模板），系统自动选定 `modular-monolith` 架构原型，生成初始蓝图与最小可运行骨架。
  - [ ] AC-005: 初始蓝图可在能力地图上完整渲染（所有原语含 `plain` 字段）。
  - [ ] AC-006: 从安装到生成可运行骨架，用户全程未见代码或终端命令。
  - [ ] AC-007: 显示执行引擎侧的订阅计划与用量信息（如 Claude Code / Codex 订阅或 API Key 的额度与消耗，数据由引擎侧接口提供）；该展示仅为执行引擎消耗透明度，与 Keel 自身商业化定价无关（定价延后，`ADR-0005`）。
- **优先级**: P0（分级依据：项目初始化是进入产品的必经路径，无此则无法创建第一个项目，所有后续功能均无从触发；且核心定位「零代码上手」要求无终端命令，初始化体验直接决定目标用户能否进门）
- **备注**: 技术栈锁定（固定托管/数据库/认证）见主卷 §4.1 产品约束。

### F-013: 意图日志与时光机

- **用户故事**: 作为用户，我希望能看到应用的演化历史，并能回到任意一个「健康」状态，无需懂 git。
- **验收标准**:
  - [ ] AC-001: 每次变更自动记录意图日志条目：原始意图文本、对应蓝图 diff、操作时间戳；「需你决策」卡片的用户决策（F-010 AC-006）同样留痕。
  - [ ] AC-002: 提供非程序员可读的演化时间线视图：以自然语言描述每次变更（「添加了用户收藏文章功能」），不暴露提交哈希、git 命令或代码差异。
  - [ ] AC-003: 任一历史变更都能查到其意图与蓝图 diff；查询响应时间 ≤ 3 秒。
  - [ ] AC-004: 时光机：用户可一键回滚到时间线上的任意点；回滚目标以自然语言描述（「回到添加收藏功能之前」），回滚后项目恢复该时间点的蓝图与代码状态，健康灯同步更新。
  - [ ] AC-005: 回滚后项目为一致状态：代码与蓝图一致，门禁可正常运行。
  - [ ] AC-006: 回滚与通道外改动的交互：回滚前若检测到时间线未记录的通道外改动（用户/第三方工具直接改文件，见 F-002 AC-006），系统须先以通俗语言提示"有没记录在时间线上的改动，回到过去会丢掉它们"，并要求用户确认后再执行回滚，不静默丢弃未留痕的用户改动。
- **优先级**: P1
- **备注**: 时间线与 git 历史的对应关系（一个意图日志条目是否对应一个 git commit）待架构阶段设计（[ASSUMPTION]）。

### F-014: 一键上线与安全体检

- **用户故事**: 作为非程序员，我希望点一下就能把应用发布给别人用，并确信系统已替我检查过安全问题，以便我不因不懂安全配置而把有漏洞的应用放到公网上。
- **验收标准**:
  - [ ] AC-001: 提供一键上线入口：触发后自动构建并部署到锁定技术栈的托管环境，完成后返回可访问地址；全程不出现终端命令或部署配置。
  - [ ] AC-002: 上线门禁（安全体检）：部署前自动检查认证配置、数据权限、密钥泄露三类安全项，以红绿灯逐项呈现并输出安全分。
  - [ ] AC-003: 存在红灯项时阻断上线，逐项给出通俗说明与修复入口；全绿后才执行部署。
  - [ ] AC-004: 上线动作记入意图日志（F-013），时间线支持回滚到上线前状态。
- **优先级**: P1
- **备注**: 三类安全项（认证配置 / 数据权限 / 密钥泄露）的最小检查清单与红灯判据见主卷 §3.2，由 **architecture 阶段**细化为具体检查器与规则集、testing/deploy 阶段验证（不存在独立的"安全规范阶段"）。托管环境为锁定技术栈的一部分（主卷 §4.1）；自建部署基础设施不在范围内。

### F-015: 开放代码与一键导出

- **用户故事**: 作为用户，我希望随时能把我的应用代码完整带走，且在没有 Keel 的情况下，代码仍能被开发者继续维护。
- **验收标准**:
  - [ ] AC-001: 随时可触发导出：生成结构服从蓝图的标准 JS/TS 仓库。
  - [ ] AC-002: 导出物含编译落地的标准工具配置（dependency-cruiser / eslint / tsconfig / JSON Schema），脱离 Keel 运行时后，这些标准工具仍能独立强制执行结构约束。
  - [ ] AC-003: 导出物为可独立运行的标准项目；未参与 Keel 的开发者可直接用标准工具（npm / node / git）继续开发。
  - [ ] AC-004: 导出物不包含 Keel 专有运行时依赖；不包含凭据或内部服务地址硬编码。
  - [ ] AC-005: 导出完成提示中说明「此项目已包含架构约束工具配置，脱离 Keel 亦可自我强制」。
- **优先级**: P1
- **备注**: 「脱离 Keel 的独立强制」的验收方式：导出后在无 Keel 环境下运行 eslint / dependency-cruiser，结构违规应被检出（[ASSUMPTION]）。

### F-016: 开发者交接包

- **用户故事**: 作为非程序员，我希望能生成一份交接文档，让日后雇来的开发者能快速看懂我的应用架构，而不需要我反复解释。
- **验收标准**:
  - [ ] AC-001: 一键生成交接包，含：蓝图转架构文档（模块、边界、数据、契约总览）、意图日志转决策史（「为什么加了这个功能」）、边界规则转上手指南（「什么能改、什么不能改」）。
  - [ ] AC-002: 交接包为独立可阅读文档（Markdown 格式），不依赖 Keel 打开。
  - [ ] AC-003: 架构文档部分可含技术视图投影（序列图、状态机图、数据流图等），面向接手开发者阅读；技术视图不进 Keel 用户主界面。
  - [ ] AC-004: 未参与项目的开发者凭交接包能在 30 分钟内定位到目标模块并找到正确修改入口（[ASSUMPTION] 具体阈值待用户研究确认，30 分钟为当前基准）；验收方式：邀请不熟悉项目的开发者实测，计时并记录是否成功定位。
- **优先级**: P2
- **备注**: 交接包内容质量依赖意图日志的记录完整性（F-013）。

### F-017: Escape Hatch（自由实现区）

- **用户故事**: 作为用户，当我的某个模块需要性能调优、复杂算法或 UI 细节等图原语无法完整表达的内容时，我希望能将其标注为「自由实现区」并附加自然语言意图说明，使引擎在此区域放宽图约束，而不影响其他模块的结构纪律。
- **验收标准**:
  - [ ] AC-001: 蓝图支持将任意 `Module` 标注为 `escape_hatch: true`，并附加 `intent`（自然语言意图说明，必填）；`intent` 为空时系统拒绝接受该标注。
  - [ ] AC-002: 标注为 escape hatch 的模块，门禁跳过依赖边界检查与架构 lint 检查（仅图结构类检查），保留接口签名校验（`Port` 契约合法性）与编译门（TS 类型检查）；编译门对 escape hatch 模块完整适用，不可豁免。显式 `Constraint.severity: hard` 的约束（含安全类规则，如密钥隔离、权限边界）不受 escape hatch 豁免，仍强制执行硬阻断；escape hatch 豁免范围仅限图结构类检查与架构 lint 软告警，与 `ADR-0004` 门禁严重度分层一致。
  - [ ] AC-003: escape hatch 模块在能力地图上以「自由实现区」标识显著区分于普通模块，并展示附加的意图说明，提示用户该区域不受图约束保护。
  - [ ] AC-004: escape hatch 标注不可传播：escape hatch 模块的依赖方仍受完整图约束；只有被显式标注的模块本身豁免检查。
  - [ ] AC-005: 导出物中 escape hatch 模块保留标注与意图说明，供接手开发者识别自由实现区边界。
- **优先级**: P1
- **备注**: escape hatch 是图原语表达力的安全阀，适用于性能调优、复杂算法、UI 细节等场景。滥用风险（将所有模块标为自由区）由产品层面的意图说明必填约束加以抑制；运营指标层面可追踪 escape hatch 占比作为架构健康信号（[ASSUMPTION]）。

### F-018: 执行引擎抽象层

- **用户故事**: 作为产品所有者，我希望 Keel 的蓝图、门禁、调和逻辑与具体执行引擎解耦，以便未来更换或扩展执行引擎不影响核心功能，也不把用户锁定在单一 AI 供应商上。
- **验收标准**:
  - [ ] AC-001: 蓝图、门禁、调和逻辑与具体执行引擎解耦；新增一个执行器实现无需改动蓝图、门禁、地图核心模块。
  - [ ] AC-002: 以 ACP（Agent Client Protocol，基于 JSON-RPC over stdio）作为标准接缝；Claude Code 与 Codex 经 adapter 接入；门禁映射为 ACP 的权限中介加编辑后检查。
  - [ ] AC-003: 原生回退到 Agent SDK 或无头 CLI，提供更完整的 hooks / subagents / sessions。
  - [ ] AC-004: 在 ACP 与原生 SDK 两种实现间切换对上层（蓝图/地图/用户界面）透明。可测判据：对**固定蓝图 + 固定一组加能力任务**，切换执行器后**蓝图产出、门禁判定结果、能力地图最终状态三者一致**（差异为空）；上层模块（蓝图/地图/用户界面）无需任何改动。
  - [ ] AC-005: 预留托管执行器接入点（为日后 SaaS 托管形态预留，当前不实现）。
  - [ ] AC-006: Keel 核心能力（门禁检查、蓝图读取/校验、能力地图状态、漂移报告）以执行引擎可直接消费的**工具面**暴露：候选形态含 MCP Server、引擎 hooks、CLI 命令，与 ACP 权限中介并列互补；具体形态组合待架构阶段选型（[ASSUMPTION]）。无论选型结果如何，执行引擎在实现/调和过程中必须能以结构化方式调用门禁并读取 JSON 结果（衔接 F-003 AC-006）。
- **优先级**: P1
- **备注**: ACP 协议的具体版本与字段映射待架构阶段确认（[ASSUMPTION]）。托管执行器接入点为架构预留，不纳入 v1 功能范围。AC-006 的工具面是**产品内部接缝**（面向执行引擎/LLM 消费），不构成面向开发者用户的独立工具产品，与 `ADR-0011` 的范围边界不冲突。
