## 问题清单

### [R-001] HIGH: 主卷 Layer 1 FAIL — 框架检查器与多卷 PRD 结构不兼容

- **category**: convention
- **root_cause**: self-caused（文档作者选择了多卷拆分结构，而未在主卷保留 AC 条目或配置检查器豁免）
- **描述**: 框架 `doc-review` Layer 1 检查器对 `volume_type=main` 的 PRD 主卷强制要求包含 `AC-NNN` 验收标准（检查逻辑：`ac_count == 0 → fail`）。`prd-keel.md` 的 `volume: main` 触发此规则，但本 PRD 的设计决策是将所有功能需求和 AC 完整下放到功能分卷，主卷只保留概述/非功能/约束/流程/术语，因此主卷合法地不含任何 AC 条目。检查器误将合理的多卷委托结构判为错误。
- **建议**: 两条可选路径（二选一）：
  1. 在主卷 `## 2. 功能需求` 章节中增加指引语句，例如在现有 `> 功能详情见分卷` 后面补充 `AC-001: （见功能分卷）`的占位引用，满足检查器的正则要求；
  2. 向框架反馈，请求 doc-review 支持 `volume_delegation: true` 类配置项豁免多卷 PRD 的跨卷 AC 检查。路径 1 为低成本修复，不依赖框架升级。

### [R-002] HIGH: PRD §1.1 能力边界未按 ADR-0009 更新"有 oracle → 远期设行为门"

- **category**: consistency
- **root_cause**: self-caused（ADR-0009 明确列出"待办：PRD G3 文案更新"，product-manager 未在 PRD 中完成此更新）
- **描述**: ADR-0009 决定的措辞调整为：从「行为一律交人工」改为「**无 oracle → 人工；有 oracle → 远期设行为门**」。当前 PRD §1.1 能力边界段落写道：「无可执行判据（oracle）的约束（产品意图、交互体验等）回退人工批准锚定。」——仅覆盖了「无 oracle → 人工」半句，缺少「有 oracle → 远期设行为门」的正向承诺。这使 PRD 与 ADR-0009 的已接受决策不一致，下游的架构师和 QA 工程师无法从 PRD 获得行为门的远期方向，可能导致架构设计时遗漏此预留点。
- **建议**: 在 PRD §1.1 「能力边界」段落追加一句：「对可写出可执行判据（oracle: test/property）的约束，远期纳入行为门（契约派生单测/属性测试）；当前阶段只做字段预留（ADR-0009）。」同时更新 F-001 AC-006 备注的措辞，与 §1.1 保持一致。

### [R-003] MEDIUM: F-003 AC-006 在产品需求层引用了框架内部工具 `cataforge skill run`

- **category**: consistency
- **root_cause**: self-caused（PRD 层混入了实现/工具细节，跨越了需求与架构边界）
- **描述**: F-003 AC-006 写道：「门禁通过 `cataforge skill run` 标准入口触发，检查结果为结构化 JSON，支持自动解析重试。」`cataforge skill run` 是 CataForge 开发框架的内部工具入口，用于 Keel 项目自身的开发流程，而非 Keel 产品向用户提供的门禁入口。将此工具名写入产品 PRD 中会造成混淆：下游架构师会不确定这里描述的是「Keel 产品对用户代码的门禁 API」还是「CataForge 框架对 Keel 项目本身的质量门禁入口」。
- **建议**: 将 AC-006 改写为描述能力意图而非实现工具，例如：「门禁通过统一的结构化接口触发，检查结果为机器可解析的 JSON 格式，支持 LLM 读取结构化错误后自动重试；具体接口协议待架构阶段确定（[ASSUMPTION]）。」如需保留 `cataforge skill run` 的关联，移入备注或约束与假设章节。

### [R-004] MEDIUM: `escape hatch`（自由实现区）在 prd-ref.md 中为核心能力，但 PRD 无对应 F-xxx 条目

- **category**: completeness
- **root_cause**: self-caused（上游需求文档中的明确功能点在 PRD 中未转化为可验证的功能条目）
- **描述**: `prd-ref.md` §3 明确列出「**表达力陷阱与逃生舱**：图原语永远无法表达全部需求（性能调优、复杂算法、UI 细节）。需要设计 escape hatch：标注为'自由实现区'的模块、附加自然语言意图说明的节点。」这是 prd-ref.md 在内核需求（约 60-70% 工程量）中单独条目列出的能力。当前 PRD 中，`escape hatch` 只出现在 §6 术语表的定义中，以及 §4.2 技术约束的间接提及（「行为门远期」上下文），但没有任何 F-xxx 功能条目定义其行为、边界或验收标准。这使 `escape hatch` 成为一个「已命名但不可验收」的概念，架构师无法据此设计，QA 工程师无法据此验证。
- **建议**: 新增 F-017（或纳入 F-001 的子项）专门描述 escape hatch 能力：支持将模块标注为「自由实现区」、附加自然语言意图说明，以及 escape hatch 不受哪些图约束检查约束；给出 1-2 条可测 AC（如「标注为 escape hatch 的模块，门禁跳过依赖边界检查，仅保留接口签名校验」）。

### [R-005] MEDIUM: 功能分卷 `deps` 字段为空，缺少对父卷的依赖声明

- **category**: convention
- **root_cause**: self-caused（分卷 front matter 约定未遵循文档引用规范）
- **描述**: `prd-keel-f001-f008.md` 和 `prd-keel-f009-f016.md` 的 front matter 均为 `deps: []`，但两个文件均声明了 `split_from: "prd-keel"`。按 COMMON-RULES §文档引用格式，`deps` 字段应列出当前文档所依赖的上游文档 id。分卷是从主卷拆分而来，逻辑上依赖 `prd-keel`；`deps: []` 使文档追溯链断裂，`cataforge docs index` 无法从分卷还原到主卷，也无法通过 `--with-deps` 展开完整 PRD 上下文。
- **建议**: 将两个分卷的 `deps: []` 改为 `deps: ["prd-keel"]`。

### [R-006] MEDIUM: F-010 回归守护与 F-003 验证门的执行边界定义不清

- **category**: ambiguity
- **root_cause**: self-caused（两个功能的职责划分在 PRD 层面未做清晰分层）
- **描述**: F-010 AC-001 写道：「随能力增加累积行为检查，每次改动后运行（**与验证门 F-003 结合执行**）」。这说明 F-010 与 F-003 共享执行路径，但 PRD 未定义两者的职责边界：F-003 处理「单次代码改动的结构合规性」，F-010 处理「跨能力的回归」——但「行为检查」究竟是 F-003 检查器的超集，还是一个独立的检查器层？两者失败时的错误消息是否相同？用户视角下是看到一条通知还是两条？架构师和实现者无法从当前 PRD 准确划分职责。
- **建议**: 在 F-010 备注或 AC-001 中补充说明执行模型分层：例如「F-003 负责本次改动的结构合规门（单点检查）；F-010 在 F-003 通过后追加跨能力回归校验（累积检查），两者串行执行，结果统一上报到能力地图健康状态。」或在 §5 关键流程 A 的步骤 6 中明确体现这一层次。

### [R-007] LOW: F-004 和 F-005 列为 P0 的必要性有待商榷

- **category**: completeness
- **root_cause**: reviewer-calibration（审查员对产品优先级的独立判断，product-manager 有解释，不构成错误）
- **描述**: 在 8 个 P0 功能中，F-001（蓝图元模型）、F-002（双向同步）、F-003（验证门）、F-006（意图捕获）、F-007（实现执行）、F-008（调和引擎）共 6 个构成核心「内核执行环」，耦合度高，P0 合理。F-004（能力地图）和 F-005（项目初始化）是非程序员产品使用路径的必要入口，但并非内核结构正确性的前提；若内核运行但地图渲染缺失，核心引擎仍可独立验证。产品上 50% P0 略超经验基准 40%，有一定风险——若 P0 全部交付才算「可演示」，则原型可用性测试（R1 核心假设验证）的门槛较高。
- **建议**: 产品团队复核 F-004 和 F-005 是否可降为 P1，同时明确「R1 可用性测试最小必要 P0 集合」。如果 F-004 是 R1 测试不可缺少的能力（非程序员必须能看到地图），则维持 P0 并在 §1.3 成功指标中显式说明「R1 测试依赖 F-004 的最小可用版本」。

### [R-008] LOW: F-013 AC-003 验收标准中的可测阈值未纳入 AC 正文

- **category**: completeness
- **root_cause**: self-caused（可测量的验收标准被放在备注而非 AC 正文，降低了可验收性）
- **描述**: F-013 AC-003 写道：「未参与项目的开发者凭交接包能在合理时间内理解项目结构并安全改动（验收方式：让不熟悉项目的开发者尝试，能找到正确的修改入口）」，但「合理时间」是模糊的，可测阈值（30 分钟定位目标模块）被放在备注的 `[ASSUMPTION]` 中。AC 正文和备注形成信息割裂，QA 工程师在验证时需要从备注中推断测试条件，容易遗漏。
- **建议**: 将阈值条件并入 AC 正文，例如改写为：「未参与项目的开发者凭交接包能在 30 分钟内定位目标模块并安全改动（[ASSUMPTION]，具体阈值待用户研究确认）；验收方式：让不熟悉项目的开发者尝试，能找到正确的修改入口。」备注保留「具体阈值待用户研究确认」即可。
