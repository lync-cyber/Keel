## Layer 2 — 两处变更语义审查

### 变更 1：F-002 AC-006 新增（来源无关性）

**变更内容**（全文）：「漂移检测**与变更来源无关**：无论代码变更经由 Keel 驱动的执行引擎产生，还是在 Keel 之外被直接修改（用户或第三方工具直接编辑文件），均被同等检测；不存在因「改动未经 Keel 通道」而漏检的路径。验收方式：在 Keel 之外直接修改文件分别制造 AC-002 四类漂移，漂移检测均须报告。（[ASSUMPTION] 外部改动的发现时机——文件系统监听实时触发，或项目打开/下次操作时扫描——待架构阶段设计；无论选型为何，不得削弱本条的来源无关语义。）」

#### 可验证性（completeness）

**验收方式可操作性**：「在 Keel 之外直接修改文件分别制造 AC-002 四类漂移，漂移检测均须报告」——验收方式与 AC-002（四类漂移：破依赖边界、重复实现同一功能、孤立未接线代码、接口契约不符）形成明确绑定，给出了具体操作步骤（外部制造四类 + 逐一验证报告），可操作。不依赖具体实现时机（文件监听 vs. 手动触发），只验证「是否报告」，属于来源无关语义的最小可测定义，合理。

**[ASSUMPTION] 标注覆盖面**：外部改动发现时机已标 [ASSUMPTION]，明确「待架构阶段设计」，且附有语义边界保障条款（「无论选型为何，不得削弱本条的来源无关语义」）。标注规范，覆盖了不确定性边界。

#### AC-006 与既有 AC 的自洽性（consistency）

**与 AC-001 的关系**：AC-001 是「代码变更后，系统能通过 AST 级静态分析……检测出漂移」。AC-001 的主语是「代码变更后」，未限定变更来源。AC-006 是对 AC-001 的语义强化：明确「代码变更后」不分 Keel 通道内外。两者语义互补，无矛盾。

**与 AC-005（增量检测 ≤ 3 秒）的关系**：这是本次审查的核心焦点之一。

- AC-005 原文：「增量检测：代码单次改动后漂移检测完成时间 ≤ 3 秒。」
- AC-006 引入了外部改动场景，其「发现时机」标注为 [ASSUMPTION]。
- 潜在歧义：若外部改动的发现时机为「文件系统监听实时触发」，则 3 秒指标可理解为「自改动被检测到起算」；若发现时机为「项目打开/下次操作时扫描」，则存在从实际改动到被检测的延迟窗口，3 秒仅覆盖「扫描执行期间」，不覆盖「静默等待期」。

**问题**：AC-005 的「代码单次改动后」语义是否被 AC-006 的外部改动场景隐式扩张？当前 AC-005 的 ≤3 秒指标对 Keel 通道内改动成立；但对于外部改动，「改动后」的计时起点尚未定义（文件被修改时刻 vs. 系统感知到修改的时刻），导致 3 秒指标在外部改动场景下的约束范围不明确。

架构师在设计文件监听方案时，若将 AC-005 解读为「自文件被修改即开始计时，含发现延迟，总计 ≤3 秒」，可能产生不必要的实现约束；若解读为「自系统感知到修改后开始计时 ≤3 秒」（含 [ASSUMPTION] 范围），则约束范围合理。此歧义应在 PRD 层明确，避免架构阶段双向误解。

#### 交叉引用正确性（consistency）

**引用 AC-002**：验收方式明确引用「AC-002 四类漂移」，AC-002 当前定义为「破依赖边界、重复实现同一功能、孤立未接线代码、接口契约不符」，引用准确。

### 变更 2：F-002 备注补充双层防御定位

**变更内容（新增部分）**：「本功能与验证门构成双层防御：F-003 在改动通道内做落地前事前拦截，F-002 对全量实际态做持续对账（reconcile loop 模式），承担通道外改动的兜底检测，检出漂移后经 F-011 调和收敛。」

#### F-003 定义核验（consistency）

**F-003 的「改动通道内事前拦截」**：F-003 AC-001 定义门禁为「编译门、契约测试门、架构 lint 门」；F-003 AC-002 为「硬违规时阻断代码落地」；F-004 AC-001 进一步说明「F-003 负责本次改动的结构合规门（单点检查）」。备注中「改动通道内做落地前事前拦截」与 F-003 的实际定义高度吻合，无矛盾。

**F-002 的「全量实际态持续对账（reconcile loop 模式）」**：F-002 AC-001 描述「通过 AST 级静态分析从代码逆向提取当前结构，并与设计蓝图持续对比」。备注中「全量实际态做持续对账」准确反映了 AC-001 的 reconcile loop 语义。

**F-011 的「调和收敛」**：F-011 标题为「调和引擎（Reconcile）」，其 AC-004 定义「调和后该项漂移消失」，AC-001/AC-003 明确调和受 F-003 门禁约束。备注中「检出漂移后经 F-011 调和收敛」与 F-011 定义一致，引用正确。

#### 双层防御逻辑自洽性（consistency）

双层防御的分工表述：
- F-003：通道内事前拦截（代码落地前检查）
- F-002：通道外改动兜底 + 全量 reconcile

F-004 AC-001 备注「F-003 负责本次改动的结构合规门（单点检查）；F-004 在 F-003 通过后追加跨能力回归校验（累积检查）」与本次 F-002 备注共同构成了三层级联。备注仅声明 F-002 与 F-003 的双层关系，未覆盖 F-004，但这是范围简洁化处理，不构成错误。

**潜在问题**：备注描述「F-003 在改动通道内做落地前事前拦截」——但 F-003 实际上是通过验证门阻断代码落地，并非「检测」已落地的漂移。与 F-002 的「检测已落地代码的漂移」在机制上有根本差异（前者是 gate/阻断，后者是 reconcile/检测）。备注使用「双层防御」框架，将「事前阻断」（F-003）与「事后检测」（F-002）并列，逻辑上可接受，但这一架构级分层叙述在 PRD 功能定义的备注区是否为恰当位置，存在轻微范畴模糊——「双层防御」是架构设计层面的表述，放入 PRD 功能需求备注可接受（属解释性说明），不构成错误，但架构文档应形成权威性阐述。

## MEDIUM 级问题

### [R-001] MEDIUM: AC-005 的 ≤3 秒指标在外部改动场景下计时起点不明确

- **category**: ambiguity
- **root_cause**: self-caused
- **描述**: AC-005（「代码单次改动后漂移检测完成时间 ≤ 3 秒」）的计时起点「改动后」对于 Keel 通道内改动是明确的（Keel 触发改动即为计时起点）；但 AC-006 引入了外部改动场景，且外部改动的发现时机已标 [ASSUMPTION]（文件系统监听 vs. 打开/操作时扫描）。当发现时机为「打开/下次操作扫描」时，AC-005 的 3 秒是否约束从「文件被外部修改的时刻」开始，还是从「系统感知到修改开始执行检测」开始，目前语义不明确。若架构师将 3 秒解读为「含发现延迟的端到端约束」，可能推导出远超合理范围的实现要求；若解读为「仅约束检测执行时间」，则 3 秒指标对外部改动场景实际上是无约束的。两种解读均可从当前文本导出，存在不一致实现风险。
- **建议**: 在 AC-005 中明确 3 秒指标的适用范围，例如改为：「增量检测：代码单次改动被系统感知后，漂移检测完成时间 ≤ 3 秒；对于外部改动，3 秒指标约束感知后的检测执行阶段，感知延迟（文件监听延迟或扫描间隔）由 [ASSUMPTION] 中的架构选型决定，不计入本指标」；或在 AC-006 的 [ASSUMPTION] 中增补「无论选型为何，3 秒指标（AC-005）仅约束从系统开始执行检测到完成报告的时间」，明确 AC-005 与 AC-006 的边界。

## LOW 级问题

### [R-002] LOW: 备注「双层防御」表述属架构级分层叙述，在 PRD 备注区存在轻微范畴模糊

- **category**: structure
- **root_cause**: self-caused
- **描述**: 备注中「双层防御」的分层描述（F-003 阻断 vs. F-002 检测/兜底）属于架构设计层面的逻辑模型，通常在架构文档中作权威性阐述。放入 PRD 功能需求备注为解释性说明，可接受，但「F-003 在改动通道内做落地前事前拦截」与「F-002 对全量实际态做持续对账」的机制差异（阻断 vs. 检测）仅凭备注文字可能被读者混淆为同等级「检测」机制，导致对 F-003 本质是门禁阻断而非漂移检测的语义误读。
- **建议**: 备注可加一句短注脚，明确 F-003 是「阻断（gate）」而非「检测（detect）」机制：「F-003 阻断结构不合规的代码落地（gate 机制），不主动检测已落地代码的漂移；F-002 的 reconcile loop 负责持续检测已落地代码与蓝图的偏差」，以强化两者的机制性区分，避免架构师在架构文档中混淆两层的职责边界。非阻塞性修改。
